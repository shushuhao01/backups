# 租户用户登录流程 — 完整方案说明

## 📋 核心结论

> **员工登录不需要输入授权码！只需输入租户编码即可。**

---

## 🎯 两种凭证的区别

| 凭证类型 | 格式 | 用途 | 谁使用 | 使用频率 |
|---------|------|------|--------|---------|
| **授权码** (License Key) | `TENANT-XXXX-XXXX-XXXX-XXXX` | 首次激活系统 | 管理员/购买人 | **仅一次** |
| **租户编码** (Tenant Code) | `T20260321XXX` | 日常登录识别企业 | 所有员工 | **每次登录** |

---

## 🔄 完整用户流程

### 场景1：客户（企业管理员）首次激活

```
1. 客户在官网购买套餐 → 收到授权码 + 登录地址
2. 打开登录页 → 选择「授权码激活」模式
3. 输入授权码 → 系统激活租户，显示企业名称
4. 输入管理员账号 admin / admin123 → 登录成功
5. 进入系统后管理自己的数据（创建用户、部门等）
```

### 场景2：企业员工日常登录

```
1. 管理员告知员工租户编码（如 T20260321XXX）
2. 员工打开登录页 → 默认「租户编码」模式
3. 输入租户编码 → 系统识别企业，显示企业名称
4. 输入自己的用户名/密码 → 登录成功
5. 浏览器记住租户编码，下次自动填充
```

### 场景3：企业员工再次登录（已缓存）

```
1. 打开登录页 → 自动显示已验证的企业信息
2. 直接输入用户名/密码 → 登录成功
3. 如需切换企业 → 点击「切换企业」按钮
```

## 🏗️ 技术架构改动

### 后端改动

1. **User 实体** (`backend/src/entities/User.ts`)
   - 新增 `tenantId` 字段（VARCHAR(36), nullable）
   - 移除 `username` 单列唯一约束（改为 tenant+username 复合唯一）

2. **JWT 配置** (`backend/src/config/jwt.ts`)
   - `JwtPayload` 接口新增 `tenantId?: string | null`

3. **登录控制器** (`backend/src/controllers/UserController.ts`)
   - `login` 方法接受 `tenantId` 参数
   - SaaS模式：`WHERE username = ? AND tenant_id = ?`
   - 私有部署：`WHERE username = ?`（不传tenantId）
   - JWT payload 中包含 `tenantId`

4. **新增API** (`backend/src/routes/tenantLicense.ts`)
   - `POST /tenant-license/verify-code` — 通过租户编码识别租户
   - 只允许已激活的租户（pending状态返回错误+自动切换到授权码模式）

5. **认证中间件** (`backend/src/middleware/auth.ts`)
   - 从JWT解析 `tenantId` 并注入 `req.tenantId`

### 前端改动

1. **登录页** (`src/views/Login.vue`)
   - 双模式切换：「租户编码」（默认） / 「授权码激活」
   - 授权验证成功后显示企业名称和套餐信息
   - `handleLogin` 传递 `tenantId` 到登录API

2. **API层** (`src/api/tenantLicense.ts`)
   - 新增 `verifyTenantCode()` 函数

3. **认证服务** (`src/services/authApiService.ts`)
   - `LoginRequest` 接口新增 `tenantId` 字段

4. **用户Store** (`src/stores/user.ts`)
   - `loginWithApi` 和 `loginWithRetry` 支持传递 `tenantId`

### 数据库改动

1. **迁移脚本** (`backend/database-migrations/add-tenant-id-to-users.sql`)
   - `users` 表添加 `tenant_id` 列
   - 添加 `idx_users_tenant_id` 索引
   - 添加 `uk_tenant_username` 复合唯一索引

## ❓ 常见问题

### Q: 如果用户只属于一个租户，能不能自动识别？
A: 可以做，但不建议。原因：
- 如果某用户名同时在多个租户中存在（如 `admin`），会产生歧义
- 统一要求输入租户编码，体验更一致，安全性更好

### Q: 租户编码会不会泄露？
A: 租户编码类似企业微信的「企业ID」，不是秘密信息。知道编码也无法登录，还需要正确的用户名密码。

### Q: 私有部署需要改什么？
A: 完全不需要改！私有部署模式下 `tenantId` 为 `undefined`，所有逻辑自动跳过租户过滤。

### Q: 已有的用户数据怎么办？
A: 
- 私有部署用户：`tenant_id = NULL`，不受影响
- SaaS租户用户：需要通过 RegisterController 或 Admin 后台创建，自动带上 `tenant_id`
