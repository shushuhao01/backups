# 模块控制功能 - 快速参考

## 一句话总结

Admin后台管理员可以启用/禁用功能模块,CRM前端菜单会自动同步,失败时自动降级保证系统可用。

## 核心特性

✅ **防御性编程** - 所有可能失败的地方都有容错  
✅ **降级策略** - API失败时使用默认列表  
✅ **缓存机制** - 5分钟缓存,减少API调用  
✅ **详细日志** - 方便排查问题  
✅ **并发控制** - 防止重复请求

## 使用流程

### Admin后台操作

1. 访问 http://localhost:5174
2. 登录 (admin / admin123)
3. 进入"模块管理"
4. 点击"启用/禁用"按钮
5. 系统管理模块不可禁用

### CRM前端效果

1. 访问 http://localhost:5173
2. 登录系统
3. 左侧菜单自动更新
4. 禁用的模块菜单被隐藏
5. dashboard始终显示

## 关键代码

### 后端API

```typescript
// backend/src/routes/system.ts
router.get('/modules/status', authenticateToken, async (_req, res) => {
  const enabledModules = await moduleRepository.find({
    where: { status: 'enabled' }
  });
  
  const menuIds = enabledModules.map(m => moduleMapping[m.code]);
  
  if (!menuIds.includes('dashboard')) {
    menuIds.unshift('dashboard');
  }
  
  res.json({ success: true, data: { enabledModules: menuIds } });
});
```

### 前端服务

```typescript
// src/services/moduleStatusService.ts
async getEnabledModules(): Promise<string[]> {
  try {
    const response = await request.get('/system/modules/status');
    return response.data.enabledModules;
  } catch (error) {
    // 失败时返回默认列表,保证系统可用
    return DEFAULT_MODULES;
  }
}
```

### 菜单过滤

```typescript
// src/utils/menu.ts
export async function getUserAccessibleMenus(menuItems: MenuItem[]) {
  // 第一步: 模块过滤
  const enabledModules = await moduleStatusService.getEnabledModules();
  const filteredByModule = filterByModuleStatus(menuItems, enabledModules);
  
  // 第二步: 权限过滤
  return filterMenuItems(filteredByModule, userRole, userPermissions);
}
```

## 容错机制

| 场景 | 处理方式 |
|------|---------|
| API调用失败 | 使用旧缓存或默认列表 |
| 数据格式异常 | 使用默认列表 |
| 并发请求 | 等待当前请求完成 |
| 缓存过期 | 自动重新获取 |

## 故障排查

### 菜单全部消失

**检查**: `moduleStatusService.ts` 的 `useDefaultModules` 方法  
**解决**: 确保返回完整的默认模块列表

### 禁用后菜单仍显示

**原因**: 缓存未刷新  
**解决**: 等待5分钟或手动刷新

```typescript
// 浏览器控制台
import { moduleStatusService } from '@/services/moduleStatusService'
await moduleStatusService.refresh()
```

### 系统管理可以禁用

**原因**: `is_system` 字段错误  
**解决**: 
```bash
cd backend
node fix-modules-system-flag.js
```

## 核心思想

**宁可多显示,不能不显示!**

所有容错机制都遵循这个原则:
- API失败 → 显示所有模块
- 数据异常 → 显示所有模块
- 缓存失效 → 显示所有模块

这样可以保证系统始终可用,不会出现"菜单全部消失"的情况。
