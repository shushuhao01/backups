# 数据隔离问题修复说明

## 问题描述

用户登录"欢乐颂"租户后，能看到"天河佳信"租户的数据（客户、订单等），说明数据隔离没有生效。

## 根本原因

虽然数据库中的数据已经正确标记了 `tenant_id`，但后端API的租户过滤没有生效。

### 技术原因

1. **数据库层面**：数据隔离正确
   - "欢乐颂"租户：0个客户、0个订单
   - "天河佳信"租户：6个客户、20个订单

2. **代码层面**：租户上下文传递断裂
   - `tenantAuth` 中间件设置了 `req.tenantId`
   - `BaseRepository` 使用 `TenantContextManager.getTenantId()` 获取租户ID
   - 但是 `tenantAuth` 没有调用 `TenantContextManager.setContext()` 更新 AsyncLocalStorage
   - 导致 `BaseRepository` 获取不到租户ID，无法过滤数据

### 执行流程

```
请求 → tenantContextMiddleware (初始化AsyncLocalStorage，但此时req.tenantId还是undefined)
     → 路由
     → tenantAuth (设置req.tenantId，但没有更新AsyncLocalStorage)
     → Controller
     → BaseRepository (从AsyncLocalStorage获取tenantId，但获取到的是undefined)
     → 查询数据库（没有tenant_id过滤，返回所有数据）
```

## 修复方案

在 `tenantAuth` 中间件中，设置完 `req.tenantId` 后，立即调用 `TenantContextManager.setContext()` 更新 AsyncLocalStorage。

### 修改文件

`backend/src/middleware/tenantAuth.ts`

### 修改内容

1. 导入 `TenantContextManager`：
```typescript
import { TenantContextManager } from '../utils/tenantContext'
```

2. 在 SaaS 模式分支中，添加上下文更新：
```typescript
// 更新 TenantContext（关键！让 BaseRepository 能获取到 tenantId）
TenantContextManager.setContext({
  tenantId: req.tenantId,
  userId: req.userId,
  tenantInfo: req.tenantInfo
})
```

3. 在私有部署模式分支中，也添加上下文更新：
```typescript
// 更新 TenantContext
TenantContextManager.setContext({
  tenantId: req.tenantId,
  userId: req.userId
})
```

## 修复后的执行流程

```
请求 → tenantContextMiddleware (初始化AsyncLocalStorage)
     → 路由
     → tenantAuth (设置req.tenantId，并调用TenantContextManager.setContext()更新AsyncLocalStorage)
     → Controller
     → BaseRepository (从AsyncLocalStorage获取tenantId，成功获取到租户ID)
     → 查询数据库（添加tenant_id过滤，只返回当前租户的数据）
```

## 相关文件

## 总结

这是一个典型的"上下文传递断裂"问题。虽然设计上使用了 AsyncLocalStorage 来传递租户上下文，但在实际执行中，`tenantAuth` 中间件没有正确更新 AsyncLocalStorage，导致 `BaseRepository` 无法获取租户ID，从而无法过滤数据。

修复后，租户数据隔离功能将正常工作，每个租户只能看到自己的数据。
