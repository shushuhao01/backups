### 2. getTenantRepo 工作原理
**文件**: `backend/src/utils/tenantRepo.ts`

```typescript
export function getTenantRepo<Entity>(entityClass: new () => Entity) {
  const repo = AppDataSource.getRepository(entityClass);
  const tenantId = getCurrentTenantId(); // 从 AsyncLocalStorage 获取
  
  // 使用 Proxy 拦截所有查询方法
  return new Proxy(repo, {
    get(target, prop) {
      // 拦截 count 方法
      if (prop === 'count') {
        return async (options?: FindManyOptions<Entity>) => {
          options = options || {};
          // ✅ 自动注入 tenant_id 过滤
          options.where = mergeTenantWhere(options.where, tenantId);
          return target.count(options);
        };
      }
      // ... 其他方法拦截
    }
  });
}
```

### 3. SQL 查询对比

**原始查询** (不安全):
```sql
SELECT COUNT(*) FROM orders WHERE status = 'pending_cancel';
-- ❌ 会返回所有租户的数据
```

**租户隔离查询** (安全):
```sql
SELECT COUNT(*) FROM orders 
WHERE tenant_id = ? AND status = 'pending_cancel';
-- ✅ 只返回当前租户的数据
```

## 租户隔离保证

### ✅ 已确保的安全措施

1. **自动注入租户ID**
   - 所有查询自动添加 `tenant_id` 过滤
   - 无需手动编写租户过滤逻辑
   - 避免人为遗漏导致的数据泄露

2. **Proxy 拦截机制**
   - 拦截所有 Repository 方法（find, count, findOne 等）
   - 在方法执行前注入租户条件
   - 无法绕过租户过滤

3. **上下文管理**
   - 使用 AsyncLocalStorage 存储租户上下文
   - 整个请求周期内租户ID保持一致
   - 不会混淆不同请求的租户信息

4. **认证中间件集成**
   - `authenticateToken` 中间件自动设置租户上下文
   - 所有需要认证的路由自动应用租户隔离
   - 未认证请求无法访问API

### ⚠️ 注意事项

1. **超级管理员场景**
   - 如果需要超管查看所有租户数据，需要特殊处理
   - 当前实现：无租户上下文时返回原始仓库（不过滤）

2. **数据库直连**
   - 直接使用 SQL 查询时需要手动添加 tenant_id 过滤
   - 建议始终使用 getTenantRepo 而非原始 Repository

3. **跨租户操作**
   - 如果业务需要跨租户查询，需要特殊设计
   - 不建议在普通业务中实现跨租户功能
