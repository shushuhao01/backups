**任务日期**：2026-03-24

**影响范围**：售后管理模块（售后订单列表、新建售后、编辑售后、售后详情、售后统计、处理人分配、优先级设置、跟进记录、操作日志、售后数据看板）

---

## 一、任务背景

CRM系统采用 SaaS 多租户架构，所有租户共享同一数据库，通过 `tenant_id` 列进行数据隔离。核心隔离工具包括：

- `getTenantRepo(Entity)` — Proxy代理仓储，自动注入 tenant_id
- `tenantSQL(prefix)` — 原始SQL的租户过滤片段
- `getCurrentTenantIdSafe()` — 获取当前请求的租户ID

本次任务深入审查售后管理模块的全部后端API接口和前端数据流，确保不同租户之间的数据完全隔离，即使不同租户存在同一客户手机号或同用户名也互不可见。

### 3.2 修复详情

#### 修复1：TENANT_ENTITIES 表名修正（tenantRepo.ts）

**修复前：**
```typescript
// 新增：服务相关
'service_follow_ups', 'service_operation_logs',
```

**修复后：**
```typescript
// 新增：服务相关（注意：ServiceFollowUp 实体表名是 service_follow_up_records）
'service_follow_up_records', 'service_operation_logs',
```

**说明**：`ServiceFollowUp` 实体的 `@Entity('service_follow_up_records')` 装饰器声明表名为 `service_follow_up_records`，而 `TENANT_ENTITIES` 中误写为 `service_follow_ups`。虽有 fallback 检查（检测 `tenantId` 列），但显式匹配更高效可靠。

#### 修复2：清理导入 + 添加验证实体（services.ts）

**修复前：**
```typescript
import { Router, Request, Response } from 'express';
import { getDataSource } from '../config/database';  // 未使用
import { AfterSalesService } from '../entities/AfterSalesService';
```

**修复后：**
```typescript
import { Router, Request, Response } from 'express';
import { AfterSalesService } from '../entities/AfterSalesService';
import { Order } from '../entities/Order';
import { Customer } from '../entities/Customer';
```

#### 修复3：创建接口添加跨租户关联验证（services.ts）

**新增代码：**
```typescript
// 🔥 租户数据隔离：验证关联订单归属当前租户
if (data.orderId) {
  const orderRepo = getTenantRepo(Order);
  const order = await orderRepo.findOne({ where: { id: data.orderId } });
  if (!order) {
    return res.status(400).json({
      success: false,
      message: '关联订单不存在或无权访问'
    });
  }
}

// 🔥 租户数据隔离：验证关联客户归属当前租户
if (data.customerId) {
  const customerRepo = getTenantRepo(Customer);
  const customer = await customerRepo.findOne({ where: { id: data.customerId } });
  if (!customer) {
    return res.status(400).json({
      success: false,
      message: '关联客户不存在或无权访问'
    });
  }
}
```

**原理**：`getTenantRepo(Order).findOne({ where: { id } })` 自动注入 `tenant_id` 条件。若 `orderId` 属于其他租户，查询为空 → 返回400 → 阻止创建。

#### 修复4：删除操作改用 delete（services.ts）

**修复前：**
```typescript
await serviceRepository.remove(service);
```

**修复后：**
```typescript
// 🔥 使用 delete 而非 remove：delete 被租户 Proxy 拦截，自动添加 tenant_id 条件，提供纵深防御
await serviceRepository.delete(id);
```

**原理**：`remove` 基于主键操作，不经过 Proxy 拦截；`delete` 经过 Proxy 拦截，会自动追加 `AND tenant_id = ?`。前置 `findOne` 已确保记录归属，此改动提供纵深防御。

## 五、数据隔离确认说明

### 5.1 同手机号不同租户场景

- 租户A的客户 `13800001234`（customerId=A001）
- 租户B的客户 `13800001234`（customerId=B001）

当租户A创建售后并传入 `customerId=B001` 时：
1. 后端调用 `getTenantRepo(Customer).findOne({ where: { id: 'B001' } })`
2. Proxy自动追加 `AND tenant_id = '租户A'`
3. 查询结果为空（B001属于租户B）
4. 返回 `400 关联客户不存在或无权访问`
5. **隔离成功** ✅

### 5.2 同用户名不同租户场景

- 租户A有用户 `zhangsan`
- 租户B有用户 `zhangsan`

分配处理人时，前端 `userStore.users` 数据来自 `GET /api/v1/users`，该接口也使用租户隔离。租户A只能看到自己的 `zhangsan`，无法看到租户B的同名用户。**隔离成功** ✅
