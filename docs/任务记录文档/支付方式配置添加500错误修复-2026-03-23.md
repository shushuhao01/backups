# 支付方式配置添加500错误修复报告

**任务日期**：2026-03-23  
**任务类型**：Bug修复  
**影响范围**：系统管理 > 系统设置 > 订单设置 > 支付方式配置  
**严重程度**：高（功能完全不可用）

---

## 一、问题描述

### 1.1 错误现象
在CRM系统「系统管理 > 系统设置 > 订单设置」标签页中，支付方式配置的**添加操作**提交失败，浏览器控制台报错：

```
POST http://localhost:5173/api/v1/system/payment-methods [HTTP/1.1 500 Internal Server Error 17ms]
```

所有支付方式相关操作（查询列表、添加、修改、删除、启用/禁用）均受影响。

### 1.2 影响范围
- **前端页面**：`src/views/Settings/OrderSettings.vue` — 支付方式配置选项卡
- **后端路由**：`backend/src/routes/system.ts` — `/payment-methods` 系列路由（GET/POST/PUT/DELETE）
- **关联页面**：`src/views/Order/List.vue`、`src/views/Order/Add.vue`、`src/views/Order/Edit.vue` 中的支付方式下拉选择

---

## 二、根因分析

### 2.1 核心原因：数据库表不存在

**实体定义** `PaymentMethodOption`（`backend/src/entities/PaymentMethodOption.ts`）已正确定义，对应数据库表 `payment_method_options`，且已在 `database.ts` 的 entities 数组中注册。

但因为 TypeORM 配置中 `synchronize: false`（第173行），数据库不会自动创建表。而该表从未被手动创建过，导致所有对 `payment_method_options` 表的查询和写入操作均抛出SQL异常。

```typescript
// backend/src/config/database.ts 第173行
synchronize: false,  // 开发环境不自动同步，避免数据丢失
```

### 2.2 辅助问题：唯一约束不适配多租户

实体定义中 `value` 列设置了 `unique: true`：

```typescript
@Column({ type: 'varchar', length: 50, unique: true, comment: '选项值' })
value: string;
```

在 SaaS 多租户模式下，不同租户可能有相同值的支付方式。唯一约束应该是 `(tenant_id, value)` 的联合唯一，而非单列唯一。

---

## 三、修复方案

### 3.1 创建缺失的数据库表

执行SQL创建 `payment_method_options` 表：

```sql
CREATE TABLE IF NOT EXISTS payment_method_options (
  id VARCHAR(50) NOT NULL,
  tenant_id VARCHAR(36) NULL COMMENT '租户ID',
  label VARCHAR(100) NOT NULL COMMENT '显示名称',
  value VARCHAR(50) NOT NULL COMMENT '选项值',
  sort_order INT NOT NULL DEFAULT 0 COMMENT '排序顺序',
  is_enabled TINYINT NOT NULL DEFAULT 1 COMMENT '是否启用',
  created_at DATETIME(6) NOT NULL DEFAULT CURRENT_TIMESTAMP(6) COMMENT '创建时间',
  updated_at DATETIME(6) NOT NULL DEFAULT CURRENT_TIMESTAMP(6) ON UPDATE CURRENT_TIMESTAMP(6) COMMENT '更新时间',
  PRIMARY KEY (id),
  UNIQUE KEY uk_tenant_value (tenant_id, value)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='支付方式选项配置表';
```

### 3.2 插入默认支付方式数据

为当前租户插入6种常用支付方式：

| ID | 显示名称 | 值 | 排序 |
|----|---------|-----|------|
| pm_default_1 | 微信支付 | wechat | 1 |
| pm_default_2 | 支付宝 | alipay | 2 |
| pm_default_3 | 银行转账 | bank_transfer | 3 |
| pm_default_4 | 现金 | cash | 4 |
| pm_default_5 | 对公转账 | bank | 5 |
| pm_default_6 | 货到付款 | cod | 6 |

### 3.3 修复实体定义中的唯一约束

**文件**：`backend/src/entities/PaymentMethodOption.ts`

```diff
- @Column({ type: 'varchar', length: 50, unique: true, comment: '选项值' })
+ @Column({ type: 'varchar', length: 50, comment: '选项值' })
  value: string;
```

移除单列 `unique` 约束，改为数据库层面的 `(tenant_id, value)` 联合唯一约束，以支持多租户场景。

## 六、后续建议

### 6.1 数据库迁移管理
建议建立数据库迁移脚本管理机制，确保新实体对应的表能被可靠地创建。当前 TypeORM `synchronize: false` 的配置下，新增实体需要手动执行建表SQL。

### 6.2 新租户初始化
建议在新租户创建时自动初始化默认支付方式数据（在租户创建流程中添加 `payment_method_options` 的初始化逻辑）。

### 6.3 建表SQL脚本维护
将本次建表SQL记录到 `database/` 目录下的迁移脚本中，方便其他环境部署时使用。
