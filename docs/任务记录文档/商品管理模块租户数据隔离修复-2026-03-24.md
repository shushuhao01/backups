### 1.2 租户隔离机制

- **核心工具**：`getTenantRepo(Entity)` — Proxy代理仓储
- **隔离方式**：共享数据库 + `tenant_id` 列隔离
- **上下文传递**：JWT → `authenticateToken` 中间件 → `TenantContextManager` (AsyncLocalStorage) → `getTenantRepo` 读取
- **自动行为**：
  - `find` / `findOne` / `count` 等查询自动添加 `tenant_id` 条件
  - `create` / `save` 自动设置 `tenant_id`
  - `createQueryBuilder` 自动添加 `WHERE tenant_id = ?`，且拦截 `.where()` 防止被覆盖

## 二、发现的问题

### 2.1 问题清单

| # | 严重程度 | 问题描述 | 位置 | 影响 |
|---|---------|---------|------|------|
| 1 | 🔴 严重 | `Product.code` 全局 `unique: true` 约束 | `entities/Product.ts:12` | 不同租户无法使用相同产品编码 |
| 2 | 🟠 高危 | 批量导入缺少编码重复校验 | `controllers/ProductController.ts:786` | 导入可能因唯一约束失败 |
| 3 | 🟡 中危 | 5处内联handler残留未使用的 `AppDataSource` 导入 | `routes/products.ts` | 未来维护可能误用绕过隔离 |
| 4 | 🟡 中危 | `ProductController` 残留未使用的 `AppDataSource, getDataSource` 导入 | `controllers/ProductController.ts:2` | 同上 |
| 5 | 🟡 中危 | 改价操作未 `await` 后端API调用 | `views/Product/List.vue:2080` | 价格修改可能仅本地未持久化 |
| 6 | 🟡 中危 | 恢复商品未调用后端API | `stores/product.ts:244` | 恢复操作不会持久化 |
| 7 | 🔵 低危 | `getProducts` 中 `if (true)` 死代码 | `controllers/ProductController.ts:339` | 代码质量问题 |

### 2.2 问题详细分析

#### 问题1：Product.code 全局唯一约束（🔴 严重）

**修复前代码**：
```typescript
// backend/src/entities/Product.ts
@Column({ type: 'varchar', length: 50, unique: true, comment: '产品编码' })
code: string
```

**问题**：`unique: true` 创建的是数据库全局唯一索引。在多租户环境下，租户A创建编码 "P001" 后，租户B也无法创建编码 "P001" 的商品，因为数据库级别的唯一约束是跨租户的。

**影响范围**：
- 新增商品时编码冲突
- 批量导入时编码冲突
- 不同租户之间不应该有编码互斥

#### 问题2：批量导入缺少编码重复校验（🟠 高危）

**修复前代码**：
```typescript
// batchImportProducts 方法中直接创建，无编码检查
const newProduct = productRepo.create({
  id: productId,
  code: productCode,
  ...
})
await productRepo.save(newProduct)
```

**问题**：单条创建 `createProduct` 有编码检查，但批量导入 `batchImportProducts` 缺少此校验。

#### 问题3-4：残留的 AppDataSource 导入（🟡 中危）

**问题**：多处代码导入了 `AppDataSource` 但实际使用的是 `getTenantRepo()`。虽然当前不构成安全问题，但后续维护者可能误用 `AppDataSource.getRepository()` 绕过租户隔离。

涉及位置：
- `routes/products.ts` — 5处内联handler（sales/statistics, sales/trend, sales/category, sales/top, inventory/warning）
- `controllers/ProductController.ts` — 顶部导入

#### 问题5：改价操作未await（🟡 中危）

```typescript
// 修复前：缺少 await，API调用可能还没完成就显示了成功
productStore.updateProduct(product.id, { price: newPrice })
ElMessage.success('改价成功')
```

#### 问题6：恢复商品未调用后端API（🟡 中危）

```typescript
// 修复前：仅修改本地Store状态，未同步后端
const restoreProduct = (id: string | number) => {
  const product = products.value.find(p => p.id === id)
  if (product && product.deleted) {
    product.deleted = false
    product.status = 'active'
    ...
  }
}
```

## 三、修复详情

### 3.1 修复1：Product实体 — 全局唯一约束改为租户级复合唯一索引

**文件**：`backend/src/entities/Product.ts`

**修复内容**：
- 移除 `code` 列的 `unique: true`
- 添加 `@Index('IDX_products_tenant_code', ['tenantId', 'code'], { unique: true })` 复合唯一索引
- 导入 TypeORM 的 `Index` 装饰器

**修复后代码**：
```typescript
import { Entity, PrimaryColumn, Column, ..., Index } from 'typeorm'

@Entity('products')
@Index('IDX_products_tenant_code', ['tenantId', 'code'], { unique: true })
export class Product {
  // ...
  @Column({ type: 'varchar', length: 50, comment: '产品编码' })
  code: string
  // ...
}
```

**效果**：不同租户可以使用相同产品编码，同一租户内编码仍保持唯一。

### 3.2 修复2：批量导入增加编码重复校验

**文件**：`backend/src/controllers/ProductController.ts`

**修复内容**：在 `batchImportProducts` 方法的循环中，创建产品前先检查编码是否已存在。

**新增代码**：
```typescript
// 🔥 修复：检查同租户内编码是否已存在，避免违反唯一约束
const existingProduct = await productRepo.findOne({ where: { code: productCode } })
if (existingProduct) {
  results.failed++
  results.errors.push(`产品 ${productCode}: 编码已存在，跳过`)
  continue
}
```

**效果**：批量导入时自动跳过重复编码的产品，并在结果中报告。

### 3.6 修复6：改价操作增加 await 确保后端同步

### 3.7 修复7：恢复商品增加后端API调用

**文件**：`src/stores/product.ts`

**修复内容**：
```typescript
// 修复前：同步方法，仅修改本地
const restoreProduct = (id: string | number) => { ... }

// 修复后：异步方法，先调用API再更新本地
const restoreProduct = async (id: string | number) => {
  const product = products.value.find(p => p.id === id)
  if (product && product.deleted) {
    try {
      await productApi.update(String(id), { status: 'active' })
    } catch (error) {
      console.error('[ProductStore] 服务器恢复商品失败:', error)
    }
    product.deleted = false
    product.status = 'active'
    product.updateTime = new Date().toISOString()
    return true
  }
  return false
}
```

## 六、注意事项

### 6.1 数据库迁移
- `Product.code` 的全局唯一索引需要在数据库中删除旧索引、创建新的复合索引
- TypeORM `synchronize: true` 模式下会自动处理
- 生产环境建议手动执行：
  ```sql
  -- 删除旧的全局唯一索引
  ALTER TABLE products DROP INDEX IDX_products_code;  -- 或 UQ_xxx 名称
  -- 创建新的复合唯一索引
  CREATE UNIQUE INDEX IDX_products_tenant_code ON products (tenant_id, code);
  ```

### 6.2 向后兼容
- `restoreProduct` 方法签名从同步改为异步（`async`），调用处如果有 `const result = restoreProduct(id)` 需要改为 `await restoreProduct(id)`
- 现有调用方 `List.vue` 中的 `handleRestore` 已经在 async 函数中调用，兼容性无问题

### 6.3 后续建议
- 考虑为 `ProductCategory` 实体也添加类似的复合唯一索引（如 `[tenantId, name, parentId]`），防止同租户同层级分类重名
- 批量导入功能可增加事务处理，确保原子性
- 库存调整记录 `getStockAdjustments` 当前返回空数据，后续需实现完整的库存变动日志记录
