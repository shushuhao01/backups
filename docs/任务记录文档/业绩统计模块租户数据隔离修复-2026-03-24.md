# 业绩统计模块租户数据隔离修复

**日期**: 2026-03-24  
**模块**: 业绩统计（Performance Statistics）  
**影响范围**: 后端 `performance.ts`、`tenantRepo.ts`  
**修复类型**: SaaS 多租户数据隔离安全修复  

---

## 一、问题分析

通过深入审查业绩统计模块的前后端代码，发现以下租户数据隔离漏洞：

### 后端 `backend/src/routes/performance.ts`

| 序号 | API 路由 | 问题描述 | 严重程度 |
|------|----------|----------|----------|
| 1 | `GET /shares/:id` | 查询 `performance_shares` 和 `performance_share_members` 时**无 tenant_id 过滤**，任何租户可查看其他租户的分享详情 | 🔴 高 |
| 2 | `POST /shares` | INSERT 到 `performance_shares` 和 `performance_share_members` 时**未写入 tenant_id 字段**，新记录无租户归属 | 🔴 高 |
| 3 | `DELETE /shares/:id` | 查询和 UPDATE 操作**无 tenant_id 过滤**，一个租户可取消另一个租户的分享记录 | 🔴 高 |
| 4 | `POST /shares/:id/confirm` | 确认分享的 UPDATE 和 COUNT 操作**无 tenant_id 过滤** | 🟡 中 |
| 5 | `GET /stats` | `userResult` JOIN 查询中 `tenantSQL('')` 无表前缀，在 JOIN 场景下 `tenant_id` 字段归属不明确 | 🟡 中 |
| 6 | `GET /personal` | 用户信息查询 `SELECT FROM users WHERE id = ?` **无 tenant_id 过滤**，可能查到其他租户的同名用户 | 🔴 高 |
| 7 | `GET /analysis/personal` | 同上，用户信息查询**无 tenant_id 过滤** | 🔴 高 |
| 8 | `GET /analysis/department` | JOIN 的 `users u` 表**无 tenant_id 过滤**，可能匹配其他租户相同 department_id 的用户 | 🔴 高 |
| 9 | `GET /analysis/metrics` (personal) | 子查询 `SELECT username FROM users WHERE id = ?` **无 tenant_id 过滤** | 🔴 高 |
| 10 | `GET /analysis/metrics` (department) | JOIN 的 `users u` 表**无 tenant_id 过滤** | 🔴 高 |

### 后端 `backend/src/utils/tenantRepo.ts`

| 序号 | 问题描述 | 严重程度 |
|------|----------|----------|
| 11 | `performance_shares` 和 `performance_share_members` **未注册**到 `TENANT_ENTITIES` 列表中，导致 `getTenantRepo()` 对这些表完全无效 | 🔴 高 |

| `GET /analysis/trend` | 已使用 `tenantSQL('')` ✅ |

### 前端分析（无需修改）

## 三、修复详情

### 修复 1：TENANT_ENTITIES 注册（tenantRepo.ts）

```typescript
// 新增：业绩分享相关
'performance_shares', 'performance_share_members',
```

将两张业绩分享表加入租户实体列表，使 `getTenantRepo()` 和 `tenantSQL()` 对这些表生效。

### 修复 2：GET /shares/:id — 添加租户隔离

```typescript
// 修复前：
const [share] = await AppDataSource.query(
  `SELECT * FROM performance_shares WHERE id = ?`, [id]
);
const members = await AppDataSource.query(
  `SELECT * FROM performance_share_members WHERE share_id = ?`, [id]
);

// 修复后：
const tDetail = tenantSQL('');
const [share] = await AppDataSource.query(
  `SELECT * FROM performance_shares WHERE id = ?${tDetail.sql}`, [id, ...tDetail.params]
);
const tMember = tenantSQL('');
const members = await AppDataSource.query(
  `SELECT * FROM performance_share_members WHERE share_id = ?${tMember.sql}`, [id, ...tMember.params]
);
```

### 修复 3：POST /shares — 写入 tenant_id

```typescript
// 修复前：INSERT 不包含 tenant_id
// 修复后：
const currentTenantId = deployConfig.isSaaS() ? TenantContextManager.getTenantId() : null;

// performance_shares INSERT 新增 tenant_id 字段
// performance_share_members INSERT 新增 tenant_id 字段
```

### 修复 4：DELETE /shares/:id — 添加租户隔离

```typescript
// 修复前：SELECT/UPDATE 无 tenant_id
// 修复后：
const tDel = tenantSQL('');
// 查询和更新均追加 ${tDel.sql}
```

### 修复 5：POST /shares/:id/confirm — 添加租户隔离

```typescript
// 修复前：UPDATE/SELECT COUNT 无 tenant_id
// 修复后：
const tConfirm = tenantSQL('');
// 所有 UPDATE 和 SELECT COUNT 均追加 ${tConfirm.sql}
```

### 修复 6：GET /stats — 修复 JOIN 查询租户前缀

```typescript
// 修复前：使用 tStats（无前缀），在 JOIN 场景下 tenant_id 字段归属不明确
// 修复后：
const tStatsUser = tenantSQL('ps.');
// userResult JOIN 查询使用 ps.tenant_id 前缀
```

### 修复 7：GET /personal — 用户查询租户隔离

```typescript
// 修复前：
const [userInfo] = await AppDataSource.query(
  `SELECT id, username FROM users WHERE id = ?`, [userId]
);

// 修复后：
const tUserLookup = tenantSQL('');
const [userInfo] = await AppDataSource.query(
  `SELECT id, username FROM users WHERE id = ?${tUserLookup.sql}`,
  [userId, ...tUserLookup.params]
);
```

### 修复 8：GET /analysis/personal — 用户查询租户隔离

```typescript
// 修复前：同修复7，无 tenant_id
// 修复后：
const tApUser = tenantSQL('');
const [userInfo] = await AppDataSource.query(
  `SELECT id, username FROM users WHERE id = ?${tApUser.sql}`,
  [userId, ...tApUser.params]
);
```

### 修复 9：GET /analysis/department — users 表 JOIN 租户隔离

```typescript
// 修复前：只对 orders 表做了隔离
const tDept = tenantSQL('o.');

// 修复后：同时对 users 表做隔离
const tDept = tenantSQL('o.');
const tDeptUser = tenantSQL('u.');
// WHERE 子句追加 ${tDept.sql}${tDeptUser.sql}
```

### 修复 10：GET /analysis/metrics — 子查询和 JOIN 租户隔离

```typescript
// 修复后（personal 类型）：
if (tMetricsUser.sql) {
  conditions.push(`(o.created_by = ? OR o.created_by = (SELECT username FROM users WHERE id = ?${tMetricsUser.sql}))`);
}

// 修复后（department 类型）：
const tMetricsJoinUser = tenantSQL('u.');
if (tMetricsJoinUser.sql) {
  conditions.push(`u.tenant_id = ?`);
}
```

## 四、安全验证

### 隔离场景覆盖

| 场景 | 验证结果 |
|------|----------|
| 不同租户有相同客户手机号 | ✅ 订单查询通过 `tenantSQL` 隔离，不会跨租户统计 |
| 不同租户有同名用户 | ✅ 用户查询添加了 `tenant_id` 过滤，不会匹配到其他租户用户 |
| 个人业绩统计 | ✅ 用户查找 + 订单查询 + 客户统计均已隔离 |
| 团队业绩统计 | ✅ 用户列表 + 订单查询均已隔离 |
| 业绩分享列表 | ✅ 已有 `tenantSQL('ps.')` |
| 业绩分享创建 | ✅ INSERT 写入 `tenant_id` |
| 业绩分享详情 | ✅ 新增 `tenantSQL('')` |
| 业绩分享取消 | ✅ 新增 `tenantSQL('')` |
| 业绩分享确认 | ✅ 新增 `tenantSQL('')` |
| 业绩分享统计 | ✅ 修复 JOIN 前缀为 `ps.` |
| 业绩分析（个人/部门/公司） | ✅ 均已做租户隔离 |
| 业绩指标（metrics） | ✅ 子查询和 JOIN 均已隔离 |
| 趋势/图表数据 | ✅ 已有 `tenantSQL` |
| 批量导出（个人/团队/分享） | ✅ 数据源头在后端，后端已隔离 |
| 分享成员选择器 | ✅ 通过 `userStore.loadUsers()` → 后端已隔离 |

### 非 SaaS 模式兼容性

- `tenantSQL()` 在非 SaaS 模式下返回 `{ sql: '', params: [] }`，不影响现有查询
- `deployConfig.isSaaS()` 为 false 时，`currentTenantId` 为 null，INSERT 的 `tenant_id` 字段值为 NULL，与原有行为一致
