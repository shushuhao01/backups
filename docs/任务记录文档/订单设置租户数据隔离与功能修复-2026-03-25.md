# 订单设置租户数据隔离与功能修复 - 2026-03-25

## 任务概述

对CRM系统"系统管理 > 系统设置 > 订单设置"选项卡中的4个问题进行排查与修复，确保多租户数据隔离正确生效。

## 问题分析与修复

### 问题1：部门下单限制配置保存失败（500错误）

**错误信息**：`POST /api/v1/system/department-order-limits 500 (Internal Server Error)`

**根因分析**：
- `department_order_limits` 数据库表在 `schema.sql` 中创建时**缺少 `tenant_id` 列**
- 但 TypeORM 实体 `DepartmentOrderLimit.ts` 中定义了 `tenantId` 字段（第8行）
- `tenantRepo.ts` 的 `TENANT_ENTITIES` 集合中包含 `department_order_limits`（第32行）
- 当 `getTenantRepo` 拦截 `save()` 操作时，会自动注入 `tenant_id` 值到 SQL INSERT 语句
- 数据库表没有该列 → SQL执行失败 → 返回500错误
- **迁移脚本遗漏**：`add-tenant-id-columns.sql` 和 `add-tenant-id-phase2.sql` 都未包含此表

**修复措施**：
1. ✅ `database/schema.sql`：`department_order_limits` 表添加 `tenant_id VARCHAR(36) NULL` 列
2. ✅ 唯一索引从 `(department_id)` 改为 `(tenant_id, department_id)` 联合唯一索引
3. ✅ 创建迁移脚本 `fix-order-settings-tenant-isolation.sql`

**配置效果确认**：  
配置生效后，部门下单限制通过 `getTenantRepo(DepartmentOrderLimit)` 自动按租户隔离：
- 管理员设置的部门限制仅对**本租户的成员和客户数据**生效
- 不同租户可以为相同 department_id 设置不同的限制配置

### 问题2：支付方式配置为空

**现象**：支付方式配置页面显示为空，预设的6种支付方式（微信、支付宝、银行转账等）不可见。

**根因分析**：
- `payment_method_options` 表的种子数据（6条预设支付方式）的 `tenant_id` 为 `NULL`
- `getTenantRepo(PaymentMethodOption)` 严格按 `tenant_id = 当前租户ID` 过滤
- `NULL ≠ 当前租户ID` → 预设数据对所有租户均不可见
- 这是数据隔离机制与全局预设数据之间的设计冲突

**修复措施**：
1. ✅ `database/schema.sql`：`payment_method_options` 表添加 `tenant_id` 列和 `is_system` 列
2. ✅ 种子数据的 `tenant_id` 设为 `NULL`，`is_system = 1`（全局预设标识）
3. ✅ 唯一索引从 `(value)` 改为 `(tenant_id, value)` 联合唯一索引
4. ✅ `PaymentMethodOption.ts` 实体添加 `isSystem` 字段
5. ✅ 后端 GET 路由改用 `AppDataSource.getRepository()` + 自定义 QueryBuilder，查询条件改为：  
   `WHERE (tenant_id = :currentTenantId OR tenant_id IS NULL)`
6. ✅ 后端 POST 路由：检查重复时同时检查全局预设和当前租户的数据
7. ✅ 后端 DELETE 路由：禁止删除系统预设（`is_system=1` 或 `tenant_id IS NULL`）
8. ✅ 前端：预设支付方式显示"预设"标签，隐藏删除按钮

**隔离效果**：
- 6种预设支付方式对**所有租户**可见（`tenant_id IS NULL`）
- 租户自己创建的支付方式只对**本租户**可见（`tenant_id = 租户ID`）
- 不同租户的自定义支付方式**互不可见**

### 问题3：订单流转时间配置隔离确认

**结论**：✅ **已正确隔离**

**分析**：
- GET/POST 路由均使用 `getTenantRepo(SystemConfig)` 操作
- `system_configs` 表在 `TENANT_ENTITIES` 集合中（第32行）
- 默认值 `mode='delayed'`, `delayMinutes=3` 是代码中的 fallback 值（第1584-1585行）
- 每个租户修改配置后，各自的 `system_configs` 记录互不影响

**附加修复**：
- ✅ `system_configs` 表的唯一索引从 `(configKey, configGroup)` 改为 `(tenant_id, configKey, configGroup)`
- **原因**：原唯一索引不含 `tenant_id`，当种子数据已存在 `(orderTransferMode, order_settings)` 记录时，其他租户创建同名配置会触发唯一约束冲突

---

### 问题4：自定义字段管理和订单来源配置隔离确认

**结论**：✅ **已正确隔离**

**分析**：
- GET/PUT 路由均使用 `getTenantRepo(SystemConfig)` 操作
- 配置键 `orderFieldConfig`（group: `order_settings`）按租户隔离存储
- 默认配置（订单来源4个选项 + 空自定义字段列表）是代码中的 fallback 值
- 每个租户可独立添加自定义字段和订单来源选项，互不影响

**受问题3附加修复保护**：
- 唯一索引修复后，不同租户可以安全创建同名配置键
