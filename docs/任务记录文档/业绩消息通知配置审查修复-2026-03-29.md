## 一、问题分析清单

### 问题1: 业绩数据不准确 — 时区问题导致订单遗漏

**影响范围**: 定时任务 + 手动测试发送 + 预览

| 编号 | 位置 | 问题描述 | 严重度 |
|------|------|---------|-------|
| 1.1 | Scheduler `checkAndSend` 73-76行 | `new Date()` 获取的是服务器本地时间，若服务器在UTC时区（如阿里云/AWS默认），`getHours()` 返回的是UTC时间而非北京时间，导致定时任务触发时间错8小时 | 🔴 严重 |
| 1.2 | Controller `generateReportData` 341-344行 | `yesterday` 和 `monthStart` 使用 `new Date()` 本地时间计算，未转换为北京时间。跨日边界（北京时间0点=UTC 16:00）的订单会被遗漏或错归日期 | 🔴 严重 |
| 1.3 | Controller `generateReportData` 354行 | `yesterday.toISOString().split('T')[0]` — `toISOString()` 返回UTC时间字符串，导致日期偏差1天 | 🔴 严重 |
| 1.4 | Controller `generateReportData` 358行 | 月统计 `o.created_at >= :start` 传入 `monthStart` Date对象，TypeORM会转为UTC datetime，导致月初数据遗漏 | 🟡 中等 |
| 1.5 | 两处 SQL 查询 | `DATE(o.created_at)` 跟随MySQL服务器时区，若MySQL在UTC时区，则 `DATE()` 提取的日期是UTC日期而非北京日期，跨午夜订单（如北京23:30=UTC15:30）日期判断错误 | 🔴 严重 |
| 1.6 | Scheduler `hasSentToday` 172-176行 | `lastSent.toDateString() === now.toDateString()` 用服务器本地时间判断，可能导致同一天重发或跳过 | 🟡 中等 |

**根因**: Controller 的 `generateReportData` 没有像 Scheduler 那样进行北京时间转换。SQL 查询中 `DATE()` 函数受 MySQL 时区影响。

## 二、修复方案

### 阶段 2: 修复时区问题 + 排行格式 + 排行数量

1. **Controller `generateReportData`**: 统一使用北京时间计算日期（参照 Scheduler 的做法）
2. **SQL查询统一**: 使用 `CONVERT_TZ(o.created_at, '+00:00', '+08:00')` 或统一用字符串日期比较 `DATE(o.created_at) = :date` 配合北京时间日期字符串
3. **Scheduler `checkAndSend`**: 统一使用北京时间判断触发和去重
4. **Markdown排行换行**: 每项排行后加 `\n\n` 确保换行
5. **排行数量**: 使用 `config.rankingLimit` 替代 hardcoded 3
6. **Scheduler 发送状态更新**: 使用 `getTenantRepo` 替代 `dataSource.getRepository`

### 阶段 3: 完善业绩类型 + 记录任务文档

1. 清理不可用的报表类型选项，保留实际可用的
2. 在消息模板中补全缺失的类型（如退款数据已查询但未展示）
3. 最终写入任务记录文档
