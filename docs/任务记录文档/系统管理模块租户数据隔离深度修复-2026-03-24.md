### 核心隔离架构

| 工具 | 说明 |
|------|------|
| `getTenantRepo(Entity)` | Proxy代理仓储，自动为 find/findOne/count/save/create/createQueryBuilder 注入 tenant_id |
| `tenantRawSQL(prefix)` | 原始SQL的租户过滤片段 `AND prefix.tenant_id = ?` |
| `getCurrentTenantIdSafe()` | 获取当前请求的租户ID（SaaS模式） |
| `TenantContextManager` | AsyncLocalStorage 存储当前请求的租户上下文 |

### 隔离原理

1. JWT 令牌中包含 `tenantId` 字段
2. `authenticateToken` 中间件从 JWT 提取 `tenantId`，存入 `TenantContextManager`
3. `getTenantRepo` 读取 `TenantContextManager` 获取 `tenantId`，通过 Proxy 拦截所有数据库操作

## 四、阶段1 — UserController 5个方法显式租户过滤修复

### 4.1 修复内容

对以下5个方法添加了与其他已有方法一致的显式租户过滤模式：

```typescript
// 统一修复模式
const tenantId = this.getTenantIdFromRequest(req);
const whereClause: any = { id: userId };
if (tenantId) {
  whereClause.tenantId = tenantId;
}
const user = await this.userRepository.findOne({ where: whereClause });
```

### 4.2 修复的方法详情

#### 1. `resetUserPassword` — 重置用户密码

**风险**：管理员可能通过构造请求，重置其他租户用户的密码  
**修复**：添加 `tenantId` 到查询条件，确保只能重置本租户用户的密码

#### 2. `forceUserLogout` — 强制用户下线

**风险**：管理员可能强制下线其他租户的用户  
**修复**：添加 `tenantId` 到查询条件

#### 3. `toggleTwoFactor` — 切换双因子认证

**风险**：管理员可能操作其他租户用户的2FA设置  
**修复**：添加 `tenantId` 到查询条件

#### 4. `unlockAccount` — 解锁用户账户

**风险**：管理员可能解锁其他租户被锁定的用户  
**修复**：添加 `tenantId` 到查询条件

#### 5. `getUserPermissions` — 获取用户权限详情

**风险**：可能泄露其他租户用户的权限配置信息  
**修复**：添加 `tenantId` 到查询条件

### 4.3 防御深度说明

虽然 `getTenantRepo(User)` 的 Proxy 已经会在 `findOne` 时自动注入 `tenantId`，但添加显式过滤提供了**防御深度**：

1. **Proxy 可能被绕过**：如果未来代码重构改变了仓储初始化方式
2. **上下文丢失风险**：AsyncLocalStorage 在某些异步场景下可能丢失上下文
3. **代码一致性**：与其他已修复方法保持统一的编码风格
4. **审计友好**：显式过滤使代码审查更容易发现租户隔离问题

### 6.2 遗留问题修复

#### 问题 1：`publishAnnouncement` 创建系统消息缺少 tenantId

**文件**：`MessageController.ts` L774  
**风险**：公告发布后创建的系统消息没有设置 `tenantId`，可能导致系统消息在跨租户场景下查询异常  
**修复**：在 `messageRepo.create()` 中添加 `tenantId: tenantId || null`

```typescript
// 修复前
const messages = targetUsers.map(user => messageRepo.create({
  id: uuidv4(),
  type: 'announcement',
  // ... 其他字段
  isRead: 0
}));

// 修复后
const messages = targetUsers.map(user => messageRepo.create({
  id: uuidv4(),
  type: 'announcement',
  // ... 其他字段
  isRead: 0,
  tenantId: tenantId || null  // 🔥 租户数据隔离
}));
```

#### 问题 2：`refreshToken` 缺少显式 tenantId 过滤

**文件**：`UserController.ts` L276-280  
**风险**：`/api/v1/auth/refresh` 接口无 `authenticateToken` 中间件 → TenantContext 未设置 → Proxy 不注入 tenant_id → 仅凭 userId 查询  
**修复**：从 JWT payload 中提取 tenantId 添加到查询条件

```typescript
// 修复前
const user = await this.userRepository.findOne({
  where: { id: payload.userId }
});

// 修复后
const refreshWhere: any = { id: payload.userId };
if (payload.tenantId) {
  refreshWhere.tenantId = payload.tenantId;
}
const user = await this.userRepository.findOne({
  where: refreshWhere
});
```

## 八、注意事项

1. **不同租户可有相同用户名/手机号/工号**：通过 `(tenant_id, username)` 复合条件实现
2. **不同租户可有相同部门名称**：`getTenantRepo` Proxy 自动限定在当前租户内
3. **角色权限仅在本租户内生效**：`RoleController` 全面使用 `tenantRawSQL()` ✅
4. **getTenantRepo Proxy 的局限**：登录和刷新令牌时无 JWT → 无 TenantContext → Proxy 返回原始仓储 → 已添加显式过滤
5. **TreeRepository 的局限**：Permission 使用 TypeORM Tree 结构 → 不支持 getTenantRepo Proxy → 已在所有方法中手动添加租户过滤
6. **仅修改指定问题**：不动其他模块代码，严格遵循禁令
7. **系统消息 tenantId**：公告发布创建的系统消息现在正确携带 tenantId，确保跨租户查询不会泄露
