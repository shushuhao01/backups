## 二、发现的问题

### 2.1 🔴 严重：租户查不到默认物流公司数据

**根因**：`logistics_companies` 表的初始数据在 `schema.sql` 中插入时 `tenant_id = NULL`，但 `getTenantRepo` 自动添加 `WHERE tenant_id = 'xxx'` 过滤，导致任何租户都查不到预设物流公司。

**影响**：
- `GET /companies/active` 返回空或500
- `GET /companies/list` 返回空或500
- 所有依赖物流公司列表的前端组件（发货弹窗、订单编辑等）降级为硬编码默认值

### 2.2 🔴 严重：数据库全局唯一约束冲突

| 表 | 列 | 问题 |
|---|---|---|
| `logistics_companies` | `code` | `UNIQUE` 全局约束，不同租户无法使用相同快递公司代码（如SF） |
| `logistics_api_configs` | `company_code` | `UNIQUE` 全局约束，不同租户无法独立配置相同快递公司API |

### 2.3 🟡 中危：前端6个组件使用 apiService.get 调用物流公司接口

**问题**：`apiService.get(url, { params })` 的 params 会被 axios 序列化为 `params[key]=value` 格式（双层嵌套），而非预期的 `key=value` 格式。应统一使用 `logisticsApi` 的封装方法。

**涉及文件**：
- `src/views/Logistics/List.vue` — `loadLogisticsCompanies()`
- `src/views/Logistics/Track.vue` — `loadLogisticsCompanies()`
- `src/views/Logistics/components/ShippingDialog.vue` — `loadLogisticsCompanies()`
- `src/views/Logistics/components/BatchShippingDialog.vue` — `loadLogisticsCompanies()`
- `src/views/Order/Add.vue` — `loadExpressCompanies()`
- `src/views/Order/Edit.vue` — `loadExpressCompanies()`

## 七、第二轮修复（2026-03-24 续）

### 7.1 🔴 发现并修复的新问题

#### 问题1：`ensureDefaultLogisticsCompanies()` 初始化逻辑缺陷

**原代码**：`if (count === 0)` — 当租户已有>=1条记录时，不再初始化其他缺失的公司

**场景**：租户首次访问时，只成功插入了德邦(DBL)，其他8家因某种原因未插入。之后每次访问检查 count=1 > 0，判定"已初始化"，导致永远只看到1个德邦

**修复**：改为逐个检查每个预设公司代码是否存在，缺失则补充插入

**修改文件**：`backend/src/routes/logistics.ts` — `ensureDefaultLogisticsCompanies()`

#### 问题2：`ensureDefaultApiConfigs()` 同样的逻辑缺陷

同上，从 `count === 0` 改为逐个 code 检查

**修改文件**：`backend/src/routes/logistics.ts` — `ensureDefaultApiConfigs()`

#### 问题3：数据库中文乱码

通过PowerShell终端直接执行的多行SQL，由于编码问题导致中文存储为 `????`

**修复方式**：使用Node.js脚本通过 `mysql2/promise` 连接（charset: 'utf8mb4'）重新插入正确数据

### 7.2 数据库修复操作

| 操作 | 说明 |
|------|------|
| 删除乱码记录 | `DELETE FROM logistics_companies WHERE tenant_id IS NOT NULL` |
| 重新插入 | 通过Node.js脚本为7个租户各插入9条正确编码的物流公司记录 |
| API配置补充 | 为7个租户各插入9条物流API配置记录（此部分中文正确，无需重做） |

### 7.3 生产环境执行
