# 订单管理模块 - 租户数据隔离修复

**任务日期**：2026-03-23  
**任务类型**：安全漏洞修复（租户数据隔离）  
**影响范围**：订单管理模块、代收管理模块、代收取消申请模块  
**严重程度**：🔴 严重 — 可导致跨租户数据泄露

## 一、问题根源

### 1.1 核心 Bug 描述

`getTenantRepo()` 的 `createQueryBuilder` 代理存在致命缺陷。代理使用 `andWhere` 注入 `tenant_id` 过滤条件：

```typescript
// tenantRepo.ts 旧代码（有缺陷）
if (prop === 'createQueryBuilder') {
    return (alias?: string) => {
        const qb = target.createQueryBuilder(alias);
        qb.andWhere(`${alias}.tenant_id = :_tid`, { _tid: tenantId });
        return qb;
    };
}
```

**但是** TypeORM 的 `.where()` 方法会 **完全替换** 之前的 WHERE 子句，包括 `andWhere` 设置的 `tenant_id` 条件。当路由代码调用：

```typescript
// 这行代码会清除掉 tenant_id 条件！
queryBuilder.where('order.status = :status', { status: 'pending_audit' });
```

**`tenant_id` 过滤条件被完全覆盖清除**，导致该查询返回所有租户的数据。

### 1.2 为什么订单列表（GET /api/v1/orders）没有问题？

主订单列表端点全程只使用 `.andWhere()`，从不使用 `.where()`：

```typescript
// 正确的写法 ✅ — 不会覆盖 tenant_id
const queryBuilder = orderRepository.createQueryBuilder('order');
queryBuilder.andWhere('order.status = :status', { status });     // andWhere ✅
queryBuilder.andWhere('order.createdBy = :userId', { userId });  // andWhere ✅
```

而有问题的端点使用了 `.where()`：

```typescript
// 错误的写法 ❌ — 会覆盖 tenant_id
queryBuilder.where('order.status = :status', { status: 'pending_audit' }); // where ❌
```

## 三、修复方案

### 3.1 根治修复：强化 tenantRepo.ts 代理（防御层 1）

**文件**：`backend/src/utils/tenantRepo.ts`

在 `createQueryBuilder` 代理中，拦截返回的 QueryBuilder 的 `.where()` 方法。当 `.where()` 被调用后，自动重新追加 `tenant_id` 条件，确保即使开发者误用 `.where()` 也不会破坏租户隔离。

```typescript
// tenantRepo.ts 新代码（已修复）
if (prop === 'createQueryBuilder') {
    return (alias?: string) => {
        const qb = target.createQueryBuilder(alias);
        const tenantCol = alias ? `${alias}.tenant_id` : `tenant_id`;

        // 设置初始租户过滤条件
        qb.andWhere(`${tenantCol} = :_tid`, { _tid: tenantId });

        // 🔥 关键修复：拦截 .where() 方法，防止被覆盖
        const originalWhere = qb.where.bind(qb);
        qb.where = function (...args: any[]) {
            originalWhere(...args);               // 先执行原始 .where()
            qb.andWhere(`${tenantCol} = :_tid`, { _tid: tenantId }); // 重新追加 tenant_id
            return qb;
        } as any;

        return qb;
    };
}
```

**原理**：
1. 代理创建 QueryBuilder 时，注入初始 `tenant_id` 条件
2. 同时拦截 `.where()` 方法
3. 当 `.where()` 被调用时，先执行原始逻辑（替换 WHERE），再自动追加 `tenant_id`
4. 这样无论代码如何使用 `.where()`，`tenant_id` 都不会丢失

### 3.2 路由修复：`.where()` → `.andWhere()`（防御层 2）

在所有受影响的路由文件中，将 `.where()` 调用改为 `.andWhere()`。即使没有代理保护，也能正确保留 `tenant_id` 条件。

**双重防护**：代理拦截 + 路由代码规范化，确保租户隔离万无一失。

## 六、开发规范建议

### 6.1 ⚠️ 使用 getTenantRepo + createQueryBuilder 时的规范

```typescript
// ✅ 正确：始终使用 andWhere
const qb = getTenantRepo(Order).createQueryBuilder('order');
qb.andWhere('order.status = :status', { status });
qb.andWhere('order.createdBy = :userId', { userId });

// ✅ 正确：链式调用也用 andWhere
const qb = getTenantRepo(Order).createQueryBuilder('order')
  .select(['order.id', 'order.orderNumber'])
  .andWhere('order.status = :status', { status: 'pending_audit' });

// ❌ 禁止：不要使用 .where()（虽然代理已防护，但编码习惯上应避免）
const qb = getTenantRepo(Order).createQueryBuilder('order');
qb.where('order.status = :status', { status });  // 不推荐
```

### 6.2 ⚠️ 代码审查检查清单

新提交的代码应检查：
1. 所有 `createQueryBuilder` 是否来自 `getTenantRepo()`（而非 `AppDataSource.getRepository()`）
2. 所有查询条件是否使用 `.andWhere()` 而非 `.where()`
3. 所有 `save()`/`create()` 操作是否通过 `getTenantRepo()` 调用
