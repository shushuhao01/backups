# 数据看板模块租户数据隔离修复

**日期**: 2026-03-24  
**模块**: 数据看板（Dashboard）  
**影响范围**: 后端 `dashboard.ts`、`MessageController.ts`  
**修复类型**: SaaS 多租户数据隔离安全修复  

---

## 一、问题分析

通过深入审查数据看板模块的前后端代码，发现以下租户数据隔离漏洞：

### 后端 `backend/src/routes/dashboard.ts`

| 序号 | API 路由 | 问题描述 | 严重程度 |
|------|----------|----------|----------|
| 1 | `GET /metrics` | 部门经理子查询 `SELECT id FROM users WHERE department_id = ?` **缺少 tenant_id 条件**，可能查到其他租户相同 department_id 的用户，导致统计数据混入其他租户的订单 | 🔴 高 |
| 2 | `GET /metrics` | 客户统计子查询 `SELECT id FROM users WHERE department_id = ?` 同样缺少租户隔离 | 🔴 高 |
| 3 | `GET /charts` | 业绩趋势、订单状态分布图表查询中部门经理子查询缺少租户隔离 | 🔴 高 |
| 4 | `GET /rankings` | 使用 `_req`（忽略请求对象），无用户角色权限过滤，任何用户都能看到全租户的排名数据 | 🟡 中 |
| 5 | `GET /todos` | 使用 `_req`（忽略请求对象），无用户角色过滤，所有人都能看到整个租户的 pending 订单 | 🟡 中 |

### 后端 `backend/src/controllers/MessageController.ts`

| 序号 | 方法 | 问题描述 | 严重程度 |
|------|------|----------|----------|
| 6 | `getSystemMessages` | 使用 `dataSource.getRepository()` 而非 `getTenantRepo()`，`createQueryBuilder` 查询缺少 `tenant_id` 条件 | 🔴 高 |
| 7 | `sendSystemMessage` | 创建消息时未设置 `tenant_id` 字段 | 🔴 高 |
| 8 | `sendBatchSystemMessages` | 批量创建消息时未设置 `tenant_id` 字段 | 🔴 高 |
| 9 | `markAllMessagesAsRead` | 查询未读消息时缺少 `tenant_id` 过滤 | 🟡 中 |
| 10 | `clearAllMessages` | 删除消息时缺少 `tenant_id` 条件，可能误删其他租户的消息 | 🔴 高 |
| 11 | `cleanupExpiredMessages` | 清理过期消息时无租户隔离，可能跨租户清理数据 | 🟡 中 |
| 12 | `deleteMessage` | 删除单条消息时缺少 `tenant_id` 条件 | 🟡 中 |
| 13 | `getMessageStats` | 消息统计缺少 `tenant_id` 过滤 | 🟡 中 |

### 前端 `src/views/Dashboard.vue` & `src/api/dashboard.ts`

前端代码主要是展示层，数据来源均为后端 API，不存在独立的数据隔离问题。快捷操作为静态数据，消息提醒通过 `notificationStore` 从后端加载。**前端无需修改**。

## 二、修复方案

### 阶段 1：后端 dashboard.ts 修复

#### 修复 1：`/metrics` 路由 - 订单查询子查询添加租户隔离
```typescript
// 修复前：
userCondition = ` AND (o.created_by IN (SELECT id FROM users WHERE department_id = ?) ...`;

// 修复后：
const tSub = tenantSQL('');
userCondition = ` AND (o.created_by IN (SELECT id FROM users WHERE department_id = ?${tSub.sql}) ...`;
params.push(departmentId, ...tSub.params, departmentId);
```

#### 修复 2：`/metrics` 路由 - 客户查询子查询添加租户隔离
```typescript
// 修复前：
customerCondition = ` AND sales_person_id IN (SELECT id FROM users WHERE department_id = ?)`;

// 修复后：
const ctSub2 = tenantSQL('');
customerCondition = ` AND sales_person_id IN (SELECT id FROM users WHERE department_id = ?${ctSub2.sql})`;
customerParams.push(departmentId, ...ctSub2.params);
```

#### 修复 3：`/charts` 路由 - 子查询添加租户隔离
与 `/metrics` 同样方式修复。

#### 修复 4：`/rankings` 路由 - 添加用户角色过滤
- `_req` → `req`，读取当前用户信息
- 非管理员只能查看本部门排名
- `getTenantRepo` 已自动处理租户级隔离

#### 修复 5：`/todos` 路由 - 添加用户角色过滤
- `_req` → `req`，读取当前用户信息
- 非管理员只能看到自己创建的待办订单
- `getTenantRepo` 已自动处理租户级隔离

### 阶段 2：MessageController.ts 修复

#### 修复 6-13：消息系统全面添加租户隔离
- 新增导入：`TenantContextManager`、`deployConfig`
- `getSystemMessages`：查询和未读统计均添加 `tenant_id` 条件
- `sendSystemMessage`：创建消息时自动设置 `tenantId`
- `sendBatchSystemMessages`：批量创建时自动设置 `tenantId`
- `markAllMessagesAsRead`：查询未读消息添加 `tenant_id` 条件
- `clearAllMessages`：删除条件添加 `tenant_id`
- `deleteMessage`：删除条件添加 `tenant_id`
- `cleanupExpiredMessages`：清理条件添加 `tenant_id`
- `getMessageStats`：统计条件添加 `tenant_id`

## 四、验证要点

### 不同租户数据隔离测试场景：
1. ✅ 租户A的管理员看不到租户B的统计数据
2. ✅ 不同租户有相同 `department_id` 时，子查询不会互相污染
3. ✅ 不同租户有同名用户时，消息不会跨租户泄露
4. ✅ 消息系统的增删改查均受 `tenant_id` 隔离保护
5. ✅ 部门经理只能看到本部门的排名，不会看到其他部门/其他租户数据
6. ✅ 普通员工只能看到个人相关的待办事项

### 前端展示验证：
- 统计汇总卡片（10个指标）：数据源自后端 `/metrics` API ✅
- 业绩趋势图：数据源自后端 `/charts` API ✅
- 订单状态分布图：数据源自后端 `/charts` API ✅
- 本月业绩排名：优先使用 `/performance/team` API（已有租户隔离），降级使用 `/rankings` API（已修复）✅
- 快捷操作：静态数据，无数据泄露风险 ✅
- 消息提醒：数据源自后端 `/message/system-messages` API（已修复）✅

## 五、注意事项

1. 所有修复均兼容私有部署模式（非SaaS场景下 `tenantSQL()` 返回空字符串，不影响查询）
2. 未修改任何前端代码，不影响前端编译和运行
3. 未修改任何不相关的代码，遵循最小修改原则
4. 原有的 ESLint 未使用变量警告（`yesterdayShippedOrders` 等）为历史遗留问题，本次不处理
