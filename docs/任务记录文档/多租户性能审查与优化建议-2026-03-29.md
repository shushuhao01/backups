## 二、数据库索引问题

### 🔴 问题 2.1：核心业务表缺失 tenant_id 列和索引

**影响等级**：严重  
**影响范围**：customers、orders、products、departments、roles 等核心表  

**问题描述**：  
`customers` 表和 `orders` 表在 `schema.sql` 中**没有 `tenant_id` 列定义**，但 TypeORM 实体（`Customer.ts`、`Order.ts`）中定义了 `tenantId` 字段。这意味着：

1. 虽然 TypeORM 可能通过同步创建了该列，但**没有索引**
2. 作为 SaaS 多租户的核心查询条件，缺少 `tenant_id` 索引会导致**全表扫描**
3. 当数据量增大时，每次查询都要遍历所有租户的数据来过滤

**涉及表**：
```
customers      - 缺少 tenant_id 列和索引 ❌
orders         - 缺少 tenant_id 列和索引 ❌  
products       - 缺少 tenant_id 列和索引 ❌
product_categories - 缺少 tenant_id 列和索引 ❌
departments    - 缺少 tenant_id 列和索引 ❌
roles          - 缺少 tenant_id 列和索引 ❌
customer_tags  - 缺少 tenant_id 列和索引 ❌
customer_groups - 缺少 tenant_id 列和索引 ❌
customer_shares - 缺少 tenant_id 列和索引 ❌
follow_up_records - 缺少 tenant_id 列和索引 ❌
call_records   - 缺少 tenant_id 列和索引 ❌
after_sales_services - 缺少 tenant_id 列和索引 ❌
service_records - 缺少 tenant_id 列和索引 ❌
logistics      - 缺少 tenant_id 列和索引 ❌
performance_records - 缺少 tenant_id 列和索引 ❌
operation_logs - 缺少 tenant_id 列和索引 ❌
data_records   - 缺少 tenant_id 列和索引 ❌
notifications  - 缺少 tenant_id 列和索引 ❌
sms_records    - 缺少 tenant_id 列和索引 ❌
sms_templates  - 缺少 tenant_id 列和索引 ❌
messages       - 缺少 tenant_id 列和索引 ❌
order_items    - 缺少 tenant_id 列和索引 ❌
order_status_history - 缺少 tenant_id 列和索引 ❌
performance_metrics - 缺少 tenant_id 列和索引 ❌
```

> ⚠️ **注意**：虽然实体中定义了 tenantId 字段，TypeORM 可能已自动同步添加了该列，但生产数据库中该列**一定没有索引**（因为 `synchronize: false`），必须手动添加。

### 🔴 问题 2.2：缺少复合索引（tenant_id + 业务字段）

**影响等级**：严重  

**问题描述**：  
即使添加了 `tenant_id` 单列索引，由于多租户查询几乎所有 WHERE 条件都包含 `tenant_id`，需要建立**复合索引**来避免回表查询。

**需要的复合索引**：
```sql
-- customers 表
(tenant_id, status)              -- 按状态查客户
(tenant_id, sales_person_id)     -- 按销售查客户
(tenant_id, created_by)          -- 按创建人查客户
(tenant_id, created_at)          -- 按时间排序
(tenant_id, phone)               -- 按手机号搜索
(tenant_id, name)                -- 按姓名搜索

-- orders 表
(tenant_id, status)              -- 按状态查订单
(tenant_id, customer_id)         -- 按客户查订单
(tenant_id, created_by)          -- 按创建人查订单
(tenant_id, created_at)          -- 按时间排序
(tenant_id, status, created_at)  -- 状态+时间复合查询

-- users 表（已有 tenant_id 索引，但缺少复合索引）
(tenant_id, department_id)       -- 按部门查用户
(tenant_id, status)              -- 按状态查用户
```

### 🟡 问题 2.3：orders 表索引过多

**影响等级**：重要  

**问题描述**：  
`orders` 表目前有 **21 个索引**，过多的索引会：
1. 严重影响写入性能（每次 INSERT/UPDATE 都要更新所有索引）
2. 占用大量磁盘空间
3. 部分索引使用频率很低（如 `idx_performance_coefficient`、`idx_cod_returned_at`）

**建议**：合并低频索引为复合索引，删除不必要的单列索引。

### 🟡 问题 2.4：customer_code 的 UNIQUE 约束未考虑租户

**影响等级**：重要  

**问题描述**：  
`customers` 表的 `customer_code` 定义为 `UNIQUE`，但不同租户之间应该允许客户编号重复。应改为 `UNIQUE(tenant_id, customer_code)` 复合唯一索引。

同样，`orders` 表的 `order_number` 也是 `UNIQUE`，需要改为租户级唯一。

### 🟡 问题 2.5：日志表缺少分区或过期清理

**影响等级**：重要  

**问题描述**：  
`operation_logs`、`logs`、`notifications` 等表会持续增长，缺少：
1. 数据归档/清理策略
2. 表分区（按月/按季度）
3. 历史数据压缩

## 三、后端N+1查询问题

### 🔴 问题 3.1：客户列表接口存在严重的 N+1 查询

**影响等级**：严重  
**文件位置**：`backend/src/routes/customers.ts` 第 222-270 行  

**问题描述**：  
客户列表接口（`GET /api/v1/customers`）在获取客户列表后，对**每一条客户记录**都执行了 3 个额外查询：

```typescript
// 🔴 对每个客户都执行以下查询（假设分页20条，则额外执行60次查询）
customers.map(async customer => {
  // 查询1：统计每个客户的订单数
  realOrderCount = await orderRepository.count({
    where: { customerId: customer.id }
  });
  
  // 查询2：查询每个客户的分享状态
  const activeShare = await shareRepository.findOne({
    where: { customerId: customer.id, status: 'active' }
  });
  
  // 查询3：查询每个客户的销售人员
  const salesPerson = await userRepository.findOne({ 
    where: { id: customer.salesPersonId } 
  });
});
```

**性能影响**：
- 分页20条时：1（主查询）+ 60（N×3子查询）= **61次数据库查询**
- 分页50条时：1 + 150 = **151次数据库查询**
- 租户越多，效果越差

**修复方案**：使用批量查询替代循环单条查询
```typescript
// ✅ 修复方案：批量获取所有需要的关联数据
const customerIds = customers.map(c => c.id);
const salesPersonIds = [...new Set(customers.map(c => c.salesPersonId).filter(Boolean))];

// 一次查询获取所有订单统计
const orderCounts = await orderRepository
  .createQueryBuilder('order')
  .select('order.customerId', 'customerId')
  .addSelect('COUNT(*)', 'count')
  .where('order.customerId IN (:...ids)', { ids: customerIds })
  .groupBy('order.customerId')
  .getRawMany();

// 一次查询获取所有分享状态
const shares = await shareRepository.find({
  where: { customerId: In(customerIds), status: 'active' }
});

// 一次查询获取所有销售人员
const salesPersons = await userRepository.find({
  where: { id: In(salesPersonIds) }
});
```

### 🔴 问题 3.2：客户列表统计查询重复且低效

**影响等级**：严重  
**文件位置**：`backend/src/routes/customers.ts` 第 193-213 行  

**问题描述**：  
为了统计"未下单客户数"，代码先查询了**所有有订单的客户ID**，再查询了**所有筛选后的客户ID**，然后在 JavaScript 中做差集：

```typescript
// 🔴 全表扫描获取有订单的客户
const customersWithOrders = await orderRepository
  .createQueryBuilder('order')
  .select('DISTINCT order.customerId', 'customerId')
  .getRawMany();

// 🔴 又查一次所有客户ID
const allFilteredCustomers = await statsQueryBuilder.clone()
  .select('customer.id')
  .getRawMany();

// 🔴 在JS中做差集（数据量大时内存消耗严重）
const noOrderCustomers = allFilteredCustomerIds.filter(
  id => !customerIdsWithOrders.includes(id)
).length;
```

**修复方案**：使用 LEFT JOIN + IS NULL 或子查询
```sql
-- ✅ 一条SQL搞定
SELECT COUNT(*) FROM customers c
LEFT JOIN orders o ON c.id = o.customer_id AND o.tenant_id = ?
WHERE c.tenant_id = ? AND o.id IS NULL
```

## 四、后端查询效率问题

### 🟡 问题 4.1：Dashboard 查询拉取全部原始数据到内存过滤

**影响等级**：重要  
**文件位置**：`backend/src/routes/dashboard.ts` 第 100-140 行  

**问题描述**：  
Dashboard 的核心指标查询将**整个时间段的所有订单**拉到内存中，然后用 JavaScript 做过滤和聚合：

```typescript
// 🔴 拉取所有今日订单到内存
const todayOrdersData = await AppDataSource.query(
  `SELECT total_amount as totalAmount, status, mark_type as markType FROM orders o WHERE ...`
);
// 🔴 在JS中过滤
const validTodayOrders = todayOrdersData.filter(o => isValidForOrderPerformance(o));
let todayOrders = validTodayOrders.length;
let todayRevenue = validTodayOrders.reduce((sum, order) => sum + Number(order.totalAmount), 0);
```

当单租户订单量大时（如月订单1万+），每次刷新仪表盘都要：
1. 拉取今日全部订单数据
2. 拉取昨日全部订单数据
3. 拉取本月全部订单数据
4. 拉取上月全部订单数据

**共4次全量查询，每次返回全部字段**。

**修复方案**：在 SQL 层完成聚合
```sql
-- ✅ 直接在数据库中聚合
SELECT 
  COUNT(*) as orderCount,
  COALESCE(SUM(total_amount), 0) as revenue
FROM orders 
WHERE tenant_id = ? 
  AND created_at >= ? AND created_at <= ?
  AND status NOT IN ('pending_cancel','cancelled','audit_rejected',
                     'logistics_returned','logistics_cancelled','refunded')
```

### 🟡 问题 4.2：hasTenantIdField() 和 getTenantFieldName() 每次都反射查询

**影响等级**：重要  
**文件位置**：`backend/src/repositories/BaseRepository.ts` 第 59-79 行  

**问题描述**：  
`BaseRepository` 的每个方法调用都会执行 `hasTenantIdField()` 和 `getTenantFieldName()`，它们每次都遍历实体的 metadata.columns 数组来查找 tenant_id 字段。虽然单次开销小，但在高并发场景下会累积。

**修复方案**：在构造函数中缓存结果
```typescript
constructor(entityClass: new () => Entity) {
  this.entityClass = entityClass;
  this.repository = AppDataSource.getRepository(entityClass);
  // ✅ 缓存租户字段信息
  this._hasTenantId = this.repository.metadata.columns.some(
    col => col.propertyName === 'tenantId' || col.propertyName === 'tenant_id'
  );
  this._tenantFieldName = /* ... */;
}
```

### 🟡 问题 4.3：订单列表查询中 ORDER BY 字段缺少索引

**影响等级**：重要  

**问题描述**：  
多处订单查询使用 `ORDER BY updatedAt DESC` 排序，但 `orders` 表只有 `created_at` 索引，没有 `updated_at` 索引。当数据量大时会导致文件排序（filesort）。

### 🟢 问题 4.4：LIKE '%keyword%' 前缀模糊查询无法走索引

**影响等级**：建议  

**问题描述**：  
多处搜索使用 `LIKE '%keyword%'`（如客户名搜索、订单搜索），这种**前后模糊匹配**无法使用索引，当数据量大时会全表扫描。

**建议**：
- 数据量不大时可以接受
- 数据量大时考虑引入全文索引（MySQL FULLTEXT）或 Elasticsearch
- 对于 phone 等精确字段，改用 `LIKE 'keyword%'`（前缀匹配可走索引）

### 🟢 问题 4.5：JSON 字段查询性能差

**问题描述**：  
客户搜索中使用 `CAST(customer.other_phones AS CHAR) LIKE :keyword`，JSON 列的 CAST 查询无法使用索引，大数据量时会很慢。

## 五、缓存机制问题

### 🔴 问题 5.1：CacheService 使用内存缓存，不支持多进程/多实例

**影响等级**：严重  
**文件位置**：`backend/src/services/CacheService.ts`  

**问题描述**：  
当前缓存服务使用 `Map` 对象做内存缓存，存在以下问题：

1. **PM2 多进程无法共享**：PM2 cluster 模式下每个进程有独立缓存，数据不一致
2. **内存泄漏风险**：没有最大缓存数量限制，高流量时可能 OOM
3. **缓存不感知租户**：缓存 key 没有包含 tenant_id，可能导致数据串租户

**修复方案**：
```typescript
// ✅ 方案1（推荐）：引入 Redis
// npm install ioredis

// ✅ 方案2（轻量）：添加 LRU 限制 + 租户感知
class CacheService {
  private cache: Map<string, CacheItem> = new Map();
  private maxSize: number = 10000; // 最大缓存条目
  
  set(key: string, data: any, ttl: number = 300): void {
    // 超过最大容量时淘汰最早的条目
    if (this.cache.size >= this.maxSize) {
      const firstKey = this.cache.keys().next().value;
      this.cache.delete(firstKey);
    }
    // ...
  }
}
```

### 🟡 问题 5.2：Dashboard 和统计数据缺少租户级缓存

**影响等级**：重要  

**问题描述**：  
`StatisticsService` 中的 dashboard 缓存 key 为 `dashboard:stats`，是全局级别的，**不区分租户**。虽然该服务主要用于 Admin 后台，但 CRM 端的 dashboard 每次请求都直接查数据库，没有使用缓存。

### 🟡 问题 5.3：认证中间件每次查询数据库

**影响等级**：重要  
**文件位置**：`backend/src/middleware/auth.ts` 第 50-53 行  

**问题描述**：  
每个 API 请求都会在认证中间件中执行一次数据库查询来获取用户信息：

```typescript
const user = await userRepository.findOne({
  where: { id: payload.userId }
});
```

**优化方案**：
- 将用户基本信息（id, role, department, status）缓存到 Redis/内存，TTL 5分钟
- JWT payload 中已包含基本信息，只在必要时（如修改密码后）才重新查库

### 🟢 问题 5.4：系统配置每次都查库

**问题描述**：  
`SystemConfig` 的查询很频繁（订单流转配置、商品设置等），但没有使用缓存。这些配置变化频率很低，适合长时间缓存。

## 六、数据库连接池问题

### 🔴 问题 6.1：连接池上限太小

**影响等级**：严重  
**文件位置**：`backend/src/config/database.ts` 第 189 行  

**问题描述**：  
```typescript
extra: {
  connectionLimit: parseInt(process.env.DB_CONNECTION_LIMIT || '10'),
}
```

默认连接池上限为 **10**，在多租户高并发场景下严重不足：
- 20个租户 × 平均3个在线用户 × 每次操作1-2连接 = 约 60-120 并发连接
- 连接池只有10个，请求会排队等待

**修复方案**：
```typescript
extra: {
  connectionLimit: parseInt(process.env.DB_CONNECTION_LIMIT || '50'),
  // 新增：连接池优化配置
  waitForConnections: true,
  queueLimit: 0,          // 不限制等待队列
  enableKeepAlive: true,   // 保持连接存活
  keepAliveInitialDelay: 10000, // 10秒
}
```

### 🟡 问题 6.2：缺少连接池监控

**影响等级**：重要  

**问题描述**：  
没有对连接池状态进行监控和告警，无法在连接耗尽前预警。

## 七、前端数据加载问题

### 🟡 问题 7.1：部分页面仍使用前端全量过滤模式

**影响等级**：重要  

**问题描述**：  
`Data/SearchNew.vue` 仍然从 store 中加载全部客户和订单数据到前端内存中，然后在前端做搜索过滤：

```typescript
const customers = customerStore.customers  // 全量客户
const orders = orderStore.getOrders()       // 全量订单
// 在前端循环过滤...
```

多租户大数据量场景下：
- 单租户5000客户 + 10000订单 → 前端内存占用约 50-100MB
- 严重影响页面响应速度和用户体验

**建议**：改为后端 API 分页搜索。

### 🟡 问题 7.2：客户列表页同时请求统计数据

**影响等级**：重要  

**问题描述**：  
客户列表接口在返回列表数据的同时，还附带了4个统计数据（总客户数、本月新增、今日新增、未下单客户），这些统计查询增加了接口响应时间。

**建议**：
1. 统计数据独立为一个接口
2. 统计数据使用缓存（30秒-1分钟）
3. 前端异步加载统计数据（先展示列表，再加载统计）

### 🟢 问题 7.3：createPersistentStore 使用 localStorage 持久化

**问题描述**：  
使用 `createPersistentStore` 将订单/客户数据持久化到 localStorage，当数据量大时：
1. localStorage 有 5MB 限制
2. 序列化/反序列化大量数据很慢
3. 多 tab 页面数据不一致

### 🟢 问题 7.4：订单详情页重复加载全部客户

**问题描述**：  
`Order/Detail.vue` 中：
```typescript
if (customerStore.customers.length === 0) {
  await customerStore.loadCustomers()  // 加载全部客户
}
```
只为显示一个客户信息，却加载全部客户。应改为按需加载单个客户。

## 八、租户数据隔离性能问题

### 🔴 问题 8.1：getTenantRepo Proxy 拦截存在性能开销

**影响等级**：严重  

**问题描述**：  
`getTenantRepo()` 每次调用都创建一个新的 `Proxy` 对象包装 Repository，在高并发场景下：
1. 频繁创建 Proxy 对象增加 GC 压力
2. Proxy 的 `get` 拦截每次属性访问都要做字符串比较
3. `createQueryBuilder` 拦截中对 `where` 方法做了二次包装

**优化方案**：
```typescript
// ✅ 使用 WeakMap 缓存已创建的代理仓储
const proxyCache = new Map<string, WeakMap<any, any>>();

export function getTenantRepo<Entity extends ObjectLiteral>(
  entityClass: new () => Entity
): Repository<Entity> {
  const tenantId = getCurrentTenantId();
  const cacheKey = `${entityClass.name}_${tenantId}`;
  
  // 同一请求中复用代理
  if (proxyCache.has(cacheKey)) {
    return proxyCache.get(cacheKey);
  }
  // ...创建代理并缓存
}
```

### 🟡 问题 8.2：tenantSQL() 构建条件时不够高效

**影响等级**：重要  

**问题描述**：  
`tenantSQL()` 每次调用都会检查 `deployConfig.isSaaS()` 和 `TenantContextManager.getTenantId()`，在一个请求处理中可能被调用 5-10 次。

**建议**：将租户上下文信息缓存到请求对象（`req.tenantId`），减少 AsyncLocalStorage 访问。

### 🟡 问题 8.3：部门经理权限查询额外开销

**影响等级**：重要  
**文件位置**：`backend/src/routes/customers.ts` 第 85-110 行  

**问题描述**：  
部门经理查看客户列表时，需要先查一次部门成员：
```typescript
const departmentMembers = await userRepository.find({
  where: { departmentId: userDepartmentId },
  select: ['id']
});
```
再将成员ID列表传入主查询的 `IN` 条件。当部门人数多或频繁查询时，这个额外查询是不必要的。

**优化**：使用子查询替代
```sql
WHERE customer.created_by IN (SELECT id FROM users WHERE department_id = ? AND tenant_id = ?)
```

  IN p_table VARCHAR(100),
  IN p_index VARCHAR(100),
  IN p_columns VARCHAR(500)
)
BEGIN
  DECLARE index_exists INT DEFAULT 0;
  SELECT COUNT(*) INTO index_exists 
  FROM information_schema.STATISTICS

    AND TABLE_NAME = p_table 
    AND INDEX_NAME = p_index;
  
  IF index_exists = 0 THEN

  END IF;
END //
