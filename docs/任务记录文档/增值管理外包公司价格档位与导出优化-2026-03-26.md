# 增值管理外包公司价格档位与导出优化

**日期**: 2026-03-26  
**模块**: 财务管理 > 增值管理 > 外包公司管理  
**类型**: 功能优化 + BUG修复

---

## 问题描述

### 1. 外包公司列表显示旧价格
- **现象**: 外包公司管理列表的"单价/比例"列始终显示旧价格档位（如¥900），而不是当前日期对应的新价格（如¥885）
- **原因**: `loadCompanies()` 中的档位匹配逻辑只按优先级排序取第一个，没有考虑档位的生效时间范围

### 2. 价格未根据订单下单时间匹配
- **现象**: 分配外包公司或同步订单时，所有订单都使用同一个固定价格，没有根据订单的下单日期来匹配不同时间段的价格档位
- **期望**: 3月22日之前的订单用旧价格（¥900），3月22日之后的订单用新价格（¥885）

### 3. 主视图单价不支持手动修改
- **现象**: 订单列表中的单价列只能显示，不支持手动修改
- **期望**: 默认使用外包公司配置的价格，同时支持手动修改输入单价

### 4. 批量导出状态显示英文
- **现象**: 导出Excel时，有效状态列显示 `valid`/`invalid`/`pending`，结算状态列显示 `settled`/`unsettled`
- **期望**: 显示中文（有效/无效/待处理、已结算/未结算）

## 修改方案

### 后端修改 (`backend/src/routes/valueAdded.ts`)

#### 1. 新增 `findPriceByOrderDate()` 辅助函数
- 根据订单下单日期匹配价格档位的核心函数
- 查找 `companyId` 对应的所有启用档位
- 将订单日期与档位的 `startDate`/`endDate` 进行比较
- 优先使用匹配到时间范围的档位，按优先级排序取第一个
- 没有匹配到日期范围时，使用最高优先级的档位作为兜底

#### 2. 修改 `syncOrdersToValueAddedOptimized()` 优化版同步函数
- 不再使用固定的 `defaultPrice`
- 每个订单根据自身的 `createdAt`（下单日期）调用 `findPriceByOrderDate()` 动态匹配
- 从 `map` 改为 `for...of` 循环（因为需要 await）

#### 3. 修改 `syncOrdersToValueAdded()` 旧版同步函数
- 同样使用 `findPriceByOrderDate()` 动态匹配价格

#### 4. 修改 `PUT /orders/:id/company` 单个订单更新公司
- 使用 `findPriceByOrderDate(companyId, order.orderDate)` 替代原来的固定优先级查找
- 保留前端提供单价时的优先使用逻辑

#### 5. 修改 `PUT /orders/batch-update-company` 批量更新公司
- 每个订单根据自身的 `orderDate` 单独匹配价格
- 不再使用统一的固定单价

#### 6. 新增 `PUT /orders/:id/unit-price` 更新单价API
- 支持手动修改单个订单的单价
- 如果订单已结算且有效，同步更新结算金额
- 更新公司统计数据

### 前端修改 (`src/views/Finance/ValueAddedManage.vue`)

#### 1. 修改 `loadCompanies()` 外包公司列表档位匹配
- 获取当前日期 `today`
- 优先匹配时间范围覆盖当前日期的档位
- 没有匹配到日期范围时，使用最高优先级的档位兜底
- 保存所有档位到 `company.allTiers` 供后续使用

#### 2. 修改单价列为可编辑

- 支持最小值0，2位小数，无步进控制按钮
- 修改后调用 `handleUnitPriceChange()` 保存到后端

#### 3. 新增 `handleUnitPriceChange()` 函数
- 调用 `updateOrderUnitPrice` API 更新单价
- 成功后更新本地数据并刷新统计

#### 4. 修改 `updateOrderCompany()` 更新公司函数
- 不再前端计算单价，让后端根据订单下单日期动态匹配
- 使用后端返回的单价更新本地数据
- 修改后自动刷新列表

#### 5. 修改 `handleBatchCompany()` 批量选择公司
- 不传单价参数，让后端为每个订单动态匹配
- 确认对话框提示：系统将根据订单下单日期自动匹配价格

#### 6. 修复批量导出状态中文化
- 新增 `getValidStatusLabel()` 函数：将有效状态值映射为中文
- 新增 `getSettlementStatusLabel()` 函数：将结算状态值映射为中文
- 优先使用动态状态配置的 label，兜底使用内置映射
- 同时修复导出的实际结算金额和结算日期格式

## 业务规则说明

### 价格档位匹配规则
1. 根据订单的**下单时间**与价格档位的**生效时间范围**（startDate ~ endDate）进行匹配
2. 多个档位匹配时，按**优先级**（priority）降序取第一个
3. 没有任何档位匹配到日期范围时，使用最高优先级的启用档位作为兜底
4. 按单计价（fixed）时使用 `unitPrice`，按比例计价（percentage）时暂不自动计算

### 单价修改规则
1. 分配外包公司时，自动根据订单下单日期匹配价格档位
2. 支持在列表中手动修改单价（覆盖自动匹配的值）
3. 如果订单已结算且有效，修改单价会同步更新结算金额

### 导出状态映射
- 有效状态：`pending` → 待处理, `valid` → 有效, `invalid` → 无效（优先使用动态配置）
- 结算状态：`unsettled` → 未结算, `settled` → 已结算（优先使用动态配置）

## 示例场景

假设外包公司配置了以下两个价格档位：
- **标准**: ¥900, 生效时间 2025-03-01 ~ 2026-03-17, 优先级 2
- **降价了**: ¥885, 生效时间 2026-03-18 ~ 2027-03-01, 优先级 1

| 订单下单时间 | 匹配档位 | 单价 |
|------------|---------|------|
| 2026-03-15 | 标准 | ¥900 |
| 2026-03-18 | 降价了 | ¥885 |
| 2026-03-22 | 降价了 | ¥885 |
| 2026-03-26 | 降价了 | ¥885 |

外包公司管理列表显示的"单价/比例"：由于当前日期 2026-03-26 在"降价了"档位的生效范围内，所以显示 ¥885。

## 注意事项

1. 已有订单的单价不会自动更新（需要重新分配公司或手动修改）
2. 新同步的订单会根据下单日期自动匹配正确的价格
3. 批量分配公司时，每个订单独立匹配价格，不再使用统一价格
4. 手动修改的单价会覆盖自动匹配的值

## 第二轮修复（2026-03-26）

### 问题5: 新增外包公司保存报错 "Cannot read properties of undefined (reading 'id')"
- **原因**: `createOutsourceCompany` 返回值经过axios拦截器已经是 `data` 字段（即company对象本身），但代码中使用了 `result.data.id`，导致 `result.data` 为 undefined
- **修复**: 改为 `const newCompany = result?.data || result`，兼容两种返回格式

### 问题6: 添加公司成功后外包公司管理列表不实时更新
- **原因**: 问题5的报错中断了后续执行，`loadCompanies()` 未被调用
- **修复**: 
  - 修复问题5后，`loadCompanies()` 可正常执行
  - 额外在外包公司管理弹窗中添加了「刷新」按钮（带loading状态），方便手动刷新

### 问题7: 价格档位排序（优先级）在公司列表显示不正确
- **原因**: 排序逻辑使用 `priority: 'DESC'`（优先级数字大的排前面），但用户期望优先级数字小的=最高优先（1最优先）
- **修复**: 
  - 后端 `findPriceByOrderDate()` 改为 `priority: 'ASC'`
  - 后端获取档位列表 API 改为 `priority: 'ASC'`
  - 前端 `loadCompanies()` 排序改为 `a.priority - b.priority`（升序）

### 问题8: 外包公司列表排序值全部显示999
- **原因**: 
  1. 实体默认值 `sortOrder = 999`
  2. 创建公司时后端未设置 `sortOrder`，使用了默认值
- **修复**:
  - 后端创建公司时自动计算下一个排序值 `MAX(sort_order) + 1`
  - 前端公司列表排序列改为 `type="index"` 动态序号（不再显示原始 sort_order 值）

## 第三轮修复（2026-03-26）：状态配置管理系统预设

### 需求描述
增值管理的状态配置管理弹窗有3个标签选项卡（有效状态、结算状态、备注预设），需要：
1. 有效状态和结算状态需要有**系统默认预设**，全员全租户生效
2. 系统预设**不可删除**
3. 用户新增的状态**仅本租户可见**，可删除
4. 备注预设可删除和新增，也仅本租户可见

### 问题分析
- 状态配置表 `value_added_status_configs` 使用 `getTenantRepo` 做租户隔离
- 新租户首次打开时，该租户下没有任何记录，所以显示空白
- 没有区分"系统预设"和"用户自定义"，所有记录都可删除

### 修改方案

#### 1. 实体层 (`backend/src/entities/ValueAddedStatusConfig.ts`)
- 新增 `is_system` 字段（TINYINT，默认0）
- 值为1表示系统预设（不可删除），值为0表示用户自定义（可删除）

#### 2. 后端 (`backend/src/routes/valueAdded.ts`)

**新增系统默认配置常量 `SYSTEM_DEFAULT_STATUS_CONFIGS`：**
- 有效状态：待处理(pending)、有效(valid)、无效(invalid)、已补单(supplemented)
- 结算状态：未结算(unsettled)、已结算(settled)

**新增 `ensureSystemDefaultConfigs()` 函数：**
- 每次查询状态配置时自动调用
- 检查当前租户是否已有系统预设（`isSystem = 1`）
- 若没有，自动初始化所有默认预设
- 已存在同 type+value 的记录会被标记为系统预设
- 自动检测并添加 `is_system` 列（防止迁移脚本未执行的情况）
- 整个函数包裹 try-catch，失败不影响正常查询

**修改 GET `/status-configs`：**
- 查询前调用 `ensureSystemDefaultConfigs()`

**修改 DELETE `/status-configs/:id`：**
- 检查 `config.isSystem === 1`，若是则返回403拒绝删除

#### 3. 前端 (`src/views/Finance/components/ValueAddedConfigDialog.vue`)
- 有效状态和结算状态的 `el-tag` 组件：
  - 系统预设（`isSystem === 1`）：不显示关闭按钮（`:closable="item.isSystem !== 1"`）
  - 系统预设显示 🔒 图标
  - 用户自定义：正常显示关闭按钮，可删除

#### 4. API接口 (`src/api/valueAdded.ts`)
- `StatusConfig` 接口新增 `isSystem?: number` 字段

#### 5. 数据库迁移脚本 (`backend/add-is-system-column.js`)
- 自动添加 `is_system` 列（如果不存在）
- 将现有的6个基础状态标记为系统预设
- 需要在服务器上手动运行：`node add-is-system-column.js`
