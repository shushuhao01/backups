## 一、项目架构概述

### 1.1 租户隔离策略
- **方案**：共享数据库 + `tenant_id` 列隔离
- **核心工具**：
  - `getTenantRepo(Entity)` — Proxy代理仓储，自动注入tenant_id到所有CRUD操作
  - `tenantSQL(prefix)` / `tenantRawSQL(prefix)` — 原始SQL的租户过滤片段
  - `addTenantFilter(qb, alias)` — QueryBuilder的租户条件注入
  - `withTenant(where)` — find/findOne的where条件注入
  - `setTenantOnEntity(entity)` — 实体保存前自动设置tenant_id
  - `TenantContextManager` — 基于AsyncLocalStorage的请求级租户上下文
  - `tenantAuth` 中间件 — 从JWT提取tenantId并设置上下文

### 1.2 问题根源
许多路由文件、控制器和服务 **直接使用** `AppDataSource.getRepository()` 或 `getDataSource().getRepository()` 绕过了租户隔离机制，导致：
- 租户A可以看到租户B的数据
- 新增数据没有自动设置`tenant_id`
- 查询/更新/删除操作没有限制在当前租户范围内

## 八、后续开发注意事项

### 8.1 开发规范
1. **禁止** 直接使用 `AppDataSource.getRepository()`，改用 `getTenantRepo()`
2. **禁止** 直接使用 `getDataSource().getRepository()`，改用 `getTenantRepo()`
3. 原始SQL查询 **必须** 使用 `tenantSQL()` 或 `tenantRawSQL()` 添加租户过滤
4. 新建实体 **必须** 包含 `tenantId` 字段（除非是全局系统表）
5. 新建表名 **必须** 注册到 `TENANT_ENTITIES` Set 中

### 8.2 代码审查检查清单
- [ ] 新路由是否使用 `getTenantRepo()` 而非 `AppDataSource.getRepository()`？
- [ ] 原始SQL是否包含 `tenantSQL()` 条件？
- [ ] QueryBuilder是否调用了 `addTenantFilter(qb, alias)`？
- [ ] 新实体是否包含 `@Column tenant_id`？
- [ ] 新表是否注册到 `TENANT_ENTITIES`？
- [ ] 关联查询的Join表是否也有tenant隔离？

### 8.3 测试建议
1. 创建两个测试租户，分别登录
2. 在租户A创建数据，验证租户B无法看到
3. 在租户B操作数据，验证不会影响租户A
4. 特别关注：物流、增值管理、消息通知、企微配置等本次修复的模块
