# 资料管理模块租户数据隔离与功能修复

**任务日期**：2026-03-24  
**任务类型**：安全修复 + 功能优化 — 租户数据隔离 & 汇总卡片 & 分页  
**影响范围**：资料管理模块（资料列表、客户查询、回收站、分配、封存、删除）

---

## 一、任务背景

CRM系统资料管理模块（`/data/*`）包含资料列表、客户查询、回收站等子功能。经全面审计发现：

1. **租户数据隔离漏洞** — 客户查询 `/search` 端点使用 `.orWhere()` 绕过了 `getTenantRepo` 的租户隔离代理，可能导致跨租户数据泄露
2. **原始SQL联表查询缺少客户表的 tenant_id 过滤** — 通过订单号/物流号搜索客户时，仅对订单表添加了 `tenant_id`，客户表未过滤
3. **汇总卡片数据不准确** — 前端从当前页数据计算统计，后端加载全部订单到内存
4. **分页参数未传递到后端** — `api.get()` 调用方式错误，导致翻页/搜索/筛选参数从未发送到后端

## 三、发现的问题

### 3.1 🔴 严重 — 租户数据隔离漏洞

| # | API路由 | 问题描述 | 严重程度 |
|---|---------|---------|----------|
| 1 | `GET /search` 客户查询 | 使用 `.where().orWhere().orWhere()` 导致后两个 OR 条件**绕过 tenant_id 隔离**。`getTenantRepo` 的 Proxy 只拦截 `.where()`，`.orWhere()` 不被拦截，SQL 变为 `WHERE (code=? AND tenant_id=?) OR phone=? OR name=?`，手机号和姓名搜索可跨租户返回数据 | 🔴 **严重** |
| 2 | `GET /search` 订单号查询 | 原始SQL `SELECT c.* FROM customers c JOIN orders o ON c.id = o.customer_id WHERE o.order_no = ?` 仅对 `o.` 添加了 `tenant_id` 过滤，`customers` 表 `c.` 未过滤 | 🔴 **严重** |
| 3 | `GET /search` 物流号查询 | 同上，`customers c` 和 `logistics_tracking l` 联表查询中客户表缺少 `tenant_id` | 🔴 **严重** |

### 3.2 🟡 中等 — 性能与逻辑问题

| # | 位置 | 问题描述 |
|---|------|---------|
| 4 | `GET /list` 汇总统计 | 使用 `orderRepository.find({ where: { status: 'delivered' } })` 加载**全部已签收订单到内存**再用JS计算统计，无权限过滤，性能低下 |
| 5 | `GET /list` 汇总权限 | 汇总统计不受角色权限限制，部门经理和销售员也能看到全租户的统计数据 |
| 6 | `GET /list` 默认分页 | 后端默认 `pageSize=30`，与前端设计的10条/页不一致 |

### 3.3 🟡 中等 — 前端问题

| # | 位置 | 问题描述 |
|---|------|---------|
| 7 | `src/api/data.ts` | `api.get('/data/list', params)` — `api.get` 期望 `{ params: {...} }` 格式，但直接传了 flat 对象，导致 `config?.params = undefined`，**所有查询参数从未发送到后端**，分页/搜索/筛选全部失效 |
| 8 | `src/views/Data/List.vue` summaryData | 汇总卡片从前端 `filteredDataList`（仅当前页数据）计算统计，数值不准 |
| 9 | `src/views/Data/List.vue` 分页 | `pageSize` ref=10，但 `onMounted` 调用 `setPagination(1, 20)`，矛盾 |
| 10 | `src/views/Data/List.vue` watch | `watch([currentPage, pageSize])` 与 `handlePageChange`/`handlePageSizeChange` 重复触发API |
| 11 | `src/stores/data.ts` | Store 默认 `pageSize=30`，与前端和后端都不一致 |

### 3.4 ✅ 已确认安全的端点

| API路由 | 隔离方式 | 状态 |
|---------|---------|------|
| `GET /list` | `getTenantRepo(Order)` + `.andWhere()` | ✅ 安全 |
| `POST /batch-assign` | `getTenantRepo(User/Customer)` | ✅ 安全 |
| `POST /batch-archive` | `getTenantRepo(Customer)` | ✅ 安全 |

## 四、修复详情

### 4.1 backend/src/routes/data.ts — 4处修复

#### 修复 1：`GET /search` — `.orWhere()` 租户隔离泄露

**修复前（🔴 危险）：**
```typescript
let customer = await customerRepository
  .createQueryBuilder('customer')
  .where('customer.customerCode = :keyword', { keyword })
  .orWhere('customer.phone = :keyword', { keyword })     // ← 绕过 tenant_id
  .orWhere('customer.name = :keyword', { keyword })       // ← 绕过 tenant_id
  .getOne();
```

生成的SQL（简化）：
```sql
WHERE (customerCode = ? AND tenant_id = ?) OR phone = ? OR name = ?
```
→ 手机号和姓名搜索**不受 tenant_id 约束**！

**修复后（✅ 安全）：**
```typescript
let customer = await customerRepository
  .createQueryBuilder('customer')
  .where('(customer.customerCode = :keyword OR customer.phone = :keyword OR customer.name = :keyword)', { keyword })
  .getOne();
```

生成的SQL：
```sql
WHERE (customerCode = ? OR phone = ? OR name = ?) AND tenant_id = ?
```
→ 所有条件都被 tenant_id AND 约束

#### 修复 2：`GET /search` 订单号查询 — 客户表缺少 tenant_id

**修复前：**
```typescript
const tData = tenantSQL('o.');
const orderResult = await AppDataSource.query(
  `SELECT c.* FROM customers c
   JOIN orders o ON c.id = o.customer_id
   WHERE o.order_no = ?${tData.sql}
   LIMIT 1`,
  [keyword, ...tData.params]
);
```

**修复后：**
```typescript
const tData = tenantSQL('o.');
const tCustomer = tenantSQL('c.');
const orderResult = await AppDataSource.query(
  `SELECT c.* FROM customers c
   JOIN orders o ON c.id = o.customer_id
   WHERE o.order_no = ?${tData.sql}${tCustomer.sql}
   LIMIT 1`,
  [keyword, ...tData.params, ...tCustomer.params]
);
```

#### 修复 3：`GET /search` 物流号查询 — 同上

增加 `const tLogisticsC = tenantSQL('c.');` 并在SQL中追加 `${tLogisticsC.sql}`。

#### 修复 4：`GET /list` 汇总统计 — 改用SQL聚合 + 权限过滤

**修复前（全量加载到内存）：**
```typescript
const allOrders = await orderRepository.find({ where: { status: 'delivered' } });
const summary = {
  totalCount: allOrders.length,
  pendingCount: allOrders.length,
  assignedCount: 0,
  totalAmount: allOrders.reduce((sum, o) => sum + Number(o.totalAmount || 0), 0),
  // ... 全部在JS中循环过滤计算
};
```

**修复后（高效SQL聚合）：**
```typescript
const summaryBuilder = orderRepository.createQueryBuilder('o')
  .select('COUNT(*)', 'totalCount')
  .addSelect('COALESCE(SUM(o.totalAmount), 0)', 'totalAmount')
  .addSelect(`SUM(CASE WHEN DATE(o.deliveredAt) = CURDATE() THEN 1 ELSE 0 END)`, 'todayCount')
  .addSelect(`SUM(CASE WHEN o.deliveredAt >= DATE_SUB(NOW(), INTERVAL 7 DAY) THEN 1 ELSE 0 END)`, 'weekCount')
  .addSelect(`SUM(CASE WHEN o.deliveredAt >= DATE_SUB(NOW(), INTERVAL 1 MONTH) THEN 1 ELSE 0 END)`, 'monthCount');
summaryBuilder.andWhere('o.status = :deliveredStatus', { deliveredStatus: 'delivered' });
// + 权限过滤（与列表查询保持一致的角色判断逻辑）
```

同时后端默认 `pageSize` 从 30 改为 10。

### 4.2 src/api/data.ts — 5处修复

| # | 函数 | 修复前 | 修复后 |
|---|------|--------|--------|
| 1 | `getDataList` | `api.get('/data/list', params)` | `api.get('/data/list', { params: params as any })` |
| 2 | `searchCustomer` | `api.get('/data/search-customer', params)` | `api.get('/data/search-customer', { params: params as any })` |
| 3 | `getDataStatistics` | `api.get('/data/statistics', { startDate, endDate })` | `api.get('/data/statistics', { params: {...} as any })` |
| 4 | `exportDataList` | `api.get('/data/export', params)` | `api.get('/data/export', { params: params as any })` |
| 5 | `getRecycleList` | `api.get('/data/recycle', params)` | `api.get('/data/recycle', { params: params as any })` |

**根本原因分析**：`api.get(endpoint, config?)` 的签名期望 `config = { params?: RequestParams }`，但调用方直接传入了 flat params 对象。由于 `config?.params` 为 `undefined`，`buildUrl()` 不会追加任何查询参数，导致翻页、搜索、日期筛选请求全部使用后端默认值。

降级分页默认值也从 30 改为 10。
