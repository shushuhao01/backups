## 二、发现的代码缺陷

### 2.1 ⚠️ 后端创建客户采用两步保存模式（严重）

**文件**：`backend/src/routes/customers.ts` - POST `/api/v1/customers`

**原始代码问题**：
```javascript
// 第一步：保存客户（无编码）
const savedCustomer = await customerRepository.save(customer);
// 第二步：生成编码并再次保存
savedCustomer.customerNo = `C${savedCustomer.id.substring(0, 8).toUpperCase()}`;
await customerRepository.save(savedCustomer); // ← 此步可能因UNIQUE约束失败
```

**风险**：
- 如果第二步保存失败（如`customer_code` UNIQUE约束冲突），客户记录已在第一步写入数据库（`customer_code = NULL`）
- API返回500错误，但数据库中已有该手机号的记录
- 下次再添加同手机号时，会提示"手机号已存在"
- 造成"有时成功有时失败"的间歇性表现

### 2.2 ⚠️ `customer_code` UNIQUE约束不含租户维度

**问题**：`customer_code` 列的UNIQUE索引是全局的，不包含 `tenant_id`。不同租户的客户可能生成相同编码前缀（基于UUID前8位），导致跨租户冲突。

### 2.3 ⚠️ 客户列表重复租户过滤

**文件**：`backend/src/routes/customers.ts` - GET `/api/v1/customers`

**问题**：`getTenantRepo(Customer)` 的Proxy已自动为 `createQueryBuilder` 添加 `tenant_id` 过滤，但代码又额外调用了 `addTenantFilter(queryBuilder, 'customer')`，造成双重过滤。虽然不会导致数据丢失，但增加不必要的复杂度。

### 2.4 ⚠️ 前端路由监听导致双重API加载

**文件**：`src/views/Customer/List.vue`

**问题**：从新增页跳转到列表页（`/customer/add` → `/customer/list?refresh=true`）时：
- **路径变化监听器**（`watch route.path`）检测到从add页返回，触发 `loadCustomerList(true)`
- **查询参数监听器**（`watch route.query`）检测到 `refresh=true`，也触发 `loadCustomerList(true)`
- 两个并发API请求互相覆盖，可能导致列表数据不稳定

### 2.5 后端日志不足

创建客户时缺少关键的租户ID、验证回查等诊断日志，难以在生产环境追踪问题根因。

## 六、后续建议

1. **修改`customer_code`唯一约束为租户级别**：
   ```sql
   ALTER TABLE customers DROP INDEX customer_code;
   ALTER TABLE customers ADD UNIQUE INDEX idx_tenant_customer_code (tenant_id, customer_code);
   ```

2. **监控后端日志**：部署后观察 `[创建客户]` 相关日志，确认创建流程完整执行

## 七、问题根因分析

### 最终结论

该问题的根因是**多个因素叠加**造成的间歇性故障：

1. **直接原因**：后端两步保存模式中，第二步（更新客户编码）可能因 `customer_code` UNIQUE约束冲突而失败，导致API返回500错误。但第一步已将客户数据写入数据库（无编码），后续添加同手机号时会被判定为"已存在"。

2. **加剧因素**：前端从新增页返回列表时，路由path和query两个watcher同时触发API请求，造成竞态条件，可能导致最终显示的数据不是最新的。

3. **混淆因素**：`getTenantRepo` Proxy与 `addTenantFilter` 的双重租户过滤虽然不会丢数据，但增加了排查难度。

### 修复后的预期效果

- ✅ 客户创建更健壮：编码生成失败不影响核心数据保存
- ✅ 列表加载更稳定：避免双重API请求竞态
- ✅ 错误提示更明确：前端能区分不同类型的失败原因
- ✅ 日志更完整：便于生产环境快速定位问题
