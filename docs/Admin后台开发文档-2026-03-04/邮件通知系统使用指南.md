# 邮件通知系统使用指南

## 一、邮件配置

### 1.1 配置位置
Admin后台 → 系统设置 → 基础配置 → 邮件配置标签页

### 1.2 配置项说明

| 配置项 | 说明 | 示例 |
|--------|------|------|
| 启用邮件 | 总开关，关闭后不发送任何邮件 | 开启/关闭 |
| SMTP服务器 | 邮件服务器地址 | smtp.qq.com |
| SMTP端口 | 服务器端口 | 465(SSL) / 587(TLS) |
| 发件人邮箱 | 系统发送邮件的邮箱 | noreply@example.com |
| 发件人名称 | 邮件显示的发件人名称 | CRM系统 |
| 邮箱密码 | SMTP授权码或密码 | QQ邮箱需使用授权码 |
| 启用SSL | 使用SSL加密 | 端口465时开启 |
| 启用TLS | 使用TLS加密 | 端口587时开启 |
| 测试邮箱 | 用于测试的收件邮箱 | test@example.com |

### 1.3 常用邮箱配置

#### QQ邮箱
```
SMTP服务器: smtp.qq.com
SMTP端口: 465
启用SSL: 是
密码: 需要在QQ邮箱设置中生成授权码
```

#### 163邮箱
```
SMTP服务器: smtp.163.com
SMTP端口: 465
启用SSL: 是
密码: 需要在163邮箱设置中开启SMTP并生成授权码
```

#### 企业邮箱
```
SMTP服务器: smtp.exmail.qq.com (腾讯企业邮箱)
SMTP端口: 465
启用SSL: 是
```

## 二、邮件触发机制

### 2.1 通知渠道服务

邮件发送通过 `NotificationChannelService` 统一管理,支持多种通知渠道:
- 邮件 (email)
- 短信 (sms)
- 钉钉 (dingtalk)
- 企业微信 (wechat_work)
- 微信公众号 (wechat_mp)
- 系统消息 (system)

### 2.2 邮件发送方式

#### 方式1: 通过通知渠道服务发送

```typescript
import { notificationChannelService } from '@/services/NotificationChannelService'

// 发送到所有启用的渠道(包括邮件)
await notificationChannelService.sendToAllChannels({
  title: '订单审核超时提醒',
  content: '订单 #12345 已超时24小时未审核',
  type: 'order_audit_timeout',
  priority: 'high',
  actionUrl: '/order/audit'
})
```

#### 方式2: 直接调用邮件服务

```typescript
import { notificationChannelService } from '@/services/NotificationChannelService'

// 只发送邮件
const channel = {
  channelType: 'email',
  config: {
    smtp_host: 'smtp.qq.com',
    smtp_port: 465,
    username: 'noreply@example.com',
    password: '授权码',
    from_name: 'CRM系统',
    to_emails: ['user1@example.com', 'user2@example.com']
  }
}

await notificationChannelService.sendToChannel(channel, {
  title: '订单审核超时提醒',
  content: '订单 #12345 已超时24小时未审核',
  actionUrl: '/order/audit'
})
```

## 四、自动触发场景

### 4.1 超时提醒服务

`TimeoutReminderService` 会自动检测并发送邮件:

| 场景 | 触发条件 | 默认超时时间 | 收件人 |
|------|----------|--------------|--------|
| 订单审核超时 | 订单状态为待审核 | 24小时 | 管理员、客服 |
| 订单发货超时 | 订单已审核但未发货 | 48小时 | 管理员、客服、下单员 |
| 售后处理超时 | 售后申请待处理 | 48小时 | 管理员、客服、处理人 |
| 订单跟进提醒 | 订单已签收 | 3天 | 下单员 |
| 客户跟进提醒 | 到达跟进计划时间 | 按计划 | 负责人 |

### 4.2 服务启动

超时提醒服务在后端启动时自动运行:

```typescript
// backend/src/app.ts
import { timeoutReminderService } from './services/TimeoutReminderService'

// 启动超时提醒服务(每30分钟检测一次)
timeoutReminderService.start(30)
```

### 4.3 配置超时时间

在Admin后台 → 系统设置 → 基础配置 → 超时提醒标签页中配置:
- 订单审核超时: 1-168小时
- 订单发货超时: 1-168小时
- 售后处理超时: 1-168小时
- 订单跟进提醒: 1-30天
- 检测间隔: 5-120分钟

## 五、手动触发邮件

### 5.1 通过Admin后台

Admin后台 → 系统设置 → 基础配置 → 超时提醒 → 点击"立即检测"按钮

### 5.2 通过API调用

```typescript
// 手动触发一次超时检测
POST /api/v1/timeout-reminder/check

// 返回结果
{
  "success": true,
  "message": "检测完成",
  "data": {
    "orderAuditTimeout": 2,      // 发送了2条订单审核超时提醒
    "orderShipmentTimeout": 1,   // 发送了1条发货超时提醒
    "afterSalesTimeout": 0,      // 没有售后超时
    "orderFollowup": 3,          // 发送了3条订单跟进提醒
    "customerFollowup": 5        // 发送了5条客户跟进提醒
  }
}
```

### 5.3 通过代码调用

```typescript
import { timeoutReminderService } from '@/services/TimeoutReminderService'

// 手动触发检测
const result = await timeoutReminderService.manualCheck()
console.log('检测结果:', result)
```

## 六、邮件发送日志

### 6.1 日志存储

所有邮件发送记录保存在 `notification_logs` 表中:

```sql
CREATE TABLE notification_logs (
  id VARCHAR(36) PRIMARY KEY,
  channel_id VARCHAR(36),           -- 渠道ID
  channel_type VARCHAR(50),         -- 渠道类型: email
  message_type VARCHAR(50),         -- 消息类型
  title VARCHAR(200),               -- 邮件标题
  content TEXT,                     -- 邮件内容
  status VARCHAR(20),               -- 状态: success/failed
  response TEXT,                    -- 发送响应
  error_message TEXT,               -- 错误信息
  sent_at DATETIME                  -- 发送时间
)
```

### 6.2 查看日志

```typescript
// 查询最近的邮件发送记录
const logs = await notificationLogRepo.find({
  where: { channelType: 'email' },
  order: { sentAt: 'DESC' },
  take: 50
})
```

## 七、常见问题

### 7.1 邮件发送失败

**问题**: 测试邮件发送失败

**解决方案**:
1. 检查SMTP服务器地址和端口是否正确
2. 确认邮箱密码是授权码而非登录密码(QQ、163邮箱)
3. 检查SSL/TLS设置是否匹配端口
4. 确认发件邮箱已开启SMTP服务
5. 查看后端日志获取详细错误信息

### 7.2 收不到邮件

**问题**: 配置正确但收不到邮件

**解决方案**:
1. 检查收件邮箱的垃圾邮件箱
2. 确认邮件服务已启用(启用邮件开关)
3. 检查通知渠道是否配置并启用
4. 查看 `notification_logs` 表确认是否发送成功
5. 检查邮箱服务商是否有发送频率限制

### 7.3 邮件内容乱码

**问题**: 邮件中文显示乱码

**解决方案**:
- 系统已自动设置 `Content-Type: text/html; charset=utf-8`
- 如仍有问题,检查邮件客户端编码设置

### 7.4 超时提醒不工作

**问题**: 配置了超时时间但不发送提醒

**解决方案**:
1. 确认超时提醒服务已启用
2. 检查后端服务是否正常运行
3. 查看后端日志确认服务是否启动
4. 手动触发检测测试功能是否正常
5. 确认邮件配置正确且已启用

## 八、最佳实践

### 8.1 邮件发送频率控制

- 避免短时间内重复发送相同提醒
- 系统已实现24小时冷却期,同一提醒24小时内只发送一次

- 标题简洁明了,包含关键信息

- 监控发送成功率

## 九、技术实现

### 9.1 核心服务

- `NotificationChannelService`: 统一通知渠道管理
- `TimeoutReminderService`: 超时检测和提醒
- `nodemailer`: Node.js邮件发送库

### 9.2 数据库表

- `system_configs`: 系统配置(邮件配置)
- `notification_channels`: 通知渠道配置
- `notification_logs`: 通知发送日志
- `system_messages`: 系统消息记录

### 9.3 API接口

```
GET  /api/v1/system/email-settings          # 获取邮件配置
PUT  /api/v1/system/email-settings          # 更新邮件配置
POST /api/v1/system/email-settings/test     # 测试邮件发送

GET  /api/v1/timeout-reminder/config        # 获取超时配置
PUT  /api/v1/timeout-reminder/config        # 更新超时配置
POST /api/v1/timeout-reminder/check         # 手动触发检测
GET  /api/v1/timeout-reminder/status        # 获取服务状态
```
