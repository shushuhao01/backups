# Admin后台生产环境数据库迁移指南

## 📋 迁移概述

本次迁移将为生产环境添加Admin后台管理系统所需的所有数据库表,包括:

- Admin后台核心表 (admin_users, admin_operation_logs)
- 租户管理表 (tenants, tenant_settings, tenant_logs)
- 套餐管理表 (packages)
- 授权管理表 (licenses, license_logs)
- 私有客户表 (private_customers)
- 版本管理表 (versions, changelogs)
- 支付订单表字段补充 (payment_orders)


## 📊 迁移内容详情

### 新增表清单

| 表名 | 说明 | 主要字段 |
|------|------|---------|
| admin_users | Admin用户表 | username, password, role, status |
| admin_operation_logs | Admin操作日志 | admin_user_id, action, module |
| tenants | 租户表 | name, code, package_id, expire_date |
| tenant_settings | 租户设置表 | tenant_id, setting_key, setting_value |
| tenant_logs | 租户日志表 | tenant_id, action, description |
| packages | 套餐表 | name, code, price, max_users |
| licenses | 授权记录表 | license_key, customer_name, status |
| license_logs | 授权日志表 | license_id, action, result |
| private_customers | 私有客户表 | customer_name, contact_person |
| versions | 版本表 | version, release_type, download_url |
| changelogs | 更新日志表 | version_id, type, content |

### 字段补充清单

**payment_orders表:**
- `billing_cycle` - 计费周期 (monthly/yearly)
- `bonus_months` - 赠送月数

**packages表:**
- `yearly_discount_rate` - 年付折扣率
- `yearly_bonus_months` - 年付赠送月数
- `yearly_price` - 年付价格

## 🔄 回滚方案

如果迁移出现问题,可以执行以下回滚操作:

### 方式一: 恢复备份

```sql

DROP TABLE IF EXISTS `admin_operation_logs`;
DROP TABLE IF EXISTS `tenants`;
DROP TABLE IF EXISTS `tenant_settings`;
DROP TABLE IF EXISTS `tenant_logs`;
DROP TABLE IF EXISTS `packages`;
DROP TABLE IF EXISTS `licenses`;
DROP TABLE IF EXISTS `license_logs`;
DROP TABLE IF EXISTS `private_customers`;
DROP TABLE IF EXISTS `versions`;
DROP TABLE IF EXISTS `changelogs`;

-- 删除payment_orders表的新增字段
ALTER TABLE `payment_orders` DROP COLUMN IF EXISTS `billing_cycle`;
ALTER TABLE `payment_orders` DROP COLUMN IF EXISTS `bonus_months`;

-- 删除packages表的新增字段
ALTER TABLE `packages` DROP COLUMN IF EXISTS `yearly_discount_rate`;
ALTER TABLE `packages` DROP COLUMN IF EXISTS `yearly_bonus_months`;
ALTER TABLE `packages` DROP COLUMN IF EXISTS `yearly_price`;
```

## 📝 注意事项

1. **执行时机**: 建议在业务低峰期执行,避免影响用户使用
2. **执行时长**: 预计执行时间 < 1分钟
3. **数据安全**: 本次迁移只创建新表和添加字段,不会修改或删除现有数据
4. **字符集**: 所有表使用 utf8mb4 字符集,支持emoji等特殊字符
5. **存储引擎**: 所有表使用 InnoDB 引擎,支持事务和外键

## 🆘 常见问题

### Q1: 执行时提示 "Table already exists"
**A:** 这是正常的,说明表已经存在。脚本使用了 `CREATE TABLE IF NOT EXISTS`,不会覆盖现有表。

### Q2: 执行时提示 "Column already exists"
**A:** 这是正常的,说明字段已经存在。脚本会检查字段是否存在再添加。

### Q3: 执行失败如何处理?
**A:** 
1. 记录错误信息
2. 检查数据库权限
3. 恢复数据库备份
4. 联系技术支持

### Q4: 如何验证迁移是否成功?
**A:** 执行上面的"执行后验证"部分的SQL语句,确认:
- 12个表都已创建
- payment_orders和packages表的新字段都已添加
- 所有索引都已创建
