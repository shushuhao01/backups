## 一、系统角色体系总览

### 1.1 数据库角色定义（`roles` 表）

| 字段 | 说明 |
|------|------|
| `id` | 角色主键 |

## 三、权限是否硬编码 - 问题分析

### 🔴 严重问题：前端权限大量硬编码，未完全从数据库获取

#### 3.1 前端 `login` 方法中的硬编码（`src/stores/user.ts`）

**问题1：本地mock登录完全硬编码**

文件 `src/stores/user.ts` 第 285-510 行，`login` 方法使用本地硬编码用户数据库：
```typescript
// 默认用户数据 - 完全硬编码
return [
  { id: 'admin', roleId: 'admin', password: 'admin123' },
  { id: 'dept_manager', roleId: 'department_manager', password: 'dept123' },
  { id: 'sales001', roleId: 'sales_staff', password: 'sales123' },
  { id: 'service001', roleId: 'customer_service', password: 'service123' }
]
```
> ⚠️ 这个 `login` 方法虽然保留了本地mock功能，但 `loginWithApi` 才是真正的API登录。两套登录逻辑并存是风险点。

**问题2：角色→权限级别映射硬编码**

文件 `src/stores/user.ts` 第 562-588 行和 649-690 行：
```typescript
// 角色到权限级别的映射完全是硬编码
role: (userRoleCode === 'admin' || userRoleCode === 'super_admin') ? 'super_admin' : ...
permissionLevel: (userRoleCode === 'admin') ? PermissionLevel.FULL_ACCESS : ...
dataScope: (userRoleCode === 'admin') ? DataScope.ALL : ...
```
> 🔴 **严重**：即使数据库中角色的 `data_scope` 是 `department`，前端也会根据角色编码硬编码覆盖。

**问题3：`permissionService.setUserPermission` 硬编码**

文件 `src/stores/user.ts` 第 649-690 行：
```typescript
if (userData.role === 'admin' || userData.role === 'super_admin') {
  permissionService.setUserPermission({
    role: UserRole.SUPER_ADMIN,
    permissions: [PermissionLevel.FULL_ACCESS],
    dataScope: DataScope.ALL
  })
} else if (userData.role === 'department_manager') {
  permissionService.setUserPermission({
    dataScope: DataScope.DEPARTMENT,
    departmentId: userData.departmentId || 'dept_001',  // 🔴 硬编码默认值
    whitelistTypes: [SensitiveInfoType.PHONE, SensitiveInfoType.EMAIL, SensitiveInfoType.WECHAT] // 🔴 硬编码
  })
}
```
> 🔴 **严重**：敏感信息访问类型、数据范围全部硬编码，不从数据库 `roles.data_scope` 或 `sensitive_info_permissions` 表读取。

#### 3.2 前端默认权限配置硬编码（`src/config/defaultRolePermissions.ts`）

整个文件定义了每个角色的默认功能权限列表，共 178 行硬编码：
```typescript
export const DEFAULT_ROLE_PERMISSIONS: Record<string, RolePermissionConfig> = {
  super_admin: { permissions: ['*'] },
  admin: { permissions: ['*'] },
  department_manager: { permissions: ['dashboard', 'customer', ...共40+项] },
  sales_staff: { permissions: ['dashboard', 'customer', ...共30+项] },
  customer_service: { permissions: ['dashboard', 'order.audit', ...共25+项] }
}
```

#### 3.3 权限加载优先级（`src/stores/user.ts` 第 590-646 行）

当前优先级：
1. **后端API返回的 `rolePermissions`**（从 `roles.permissions` JSON字段读取） ✅
2. **localStorage `crm_roles` 缓存** ⚠️
3. **`defaultRolePermissions.ts` 硬编码配置** 🔴
4. **角色名称映射后的默认配置** 🔴

> 结论：**只有第1步是从数据库读取的**。如果后端未返回权限（如新角色），则回退到硬编码。

#### 3.4 前端 `PermissionService` 初始化硬编码（`src/services/permission.ts`）

文件第 104-222 行，`initializeDefaultPermissions` 方法硬编码了8个用户的权限：
```typescript
this.userPermissions.set('admin', { role: UserRole.SUPER_ADMIN, ... })
this.userPermissions.set('manager', { role: UserRole.DEPARTMENT_MANAGER, ... })
this.userPermissions.set('sales001', { ... })
// ... 共8个硬编码用户
```
> 🔴 **严重**：这些是按 `userId` 硬编码的，只对特定测试用户有效。实际用户需要通过 `setUserPermission` 动态设置。

#### 3.5 前端路由守卫硬编码角色检查（`src/router/index.ts`）

路由 `meta` 中使用硬编码的角色标记：

而 `isAdmin` 的判断也是硬编码（第 115-118 行）：

> 🔴 如果创建新的管理员角色（如 `branch_admin`），前端无法识别。

#### 3.6 后端角色硬编码（`backend/src/middleware/auth.ts`）


> ⚠️ 系统保护角色硬编码。

## 四、租户数据隔离 - 全面审计

### 4.1 租户隔离机制概述

系统使用**共享数据库 + `tenant_id` 字段**的隔离策略：
- 所有需要隔离的表都有 `tenant_id` 列
- 通过 `getTenantRepo()` Proxy 包装自动注入过滤条件
- 通过 `tenantHelpers.ts` 提供辅助函数
- SaaS模式通过 `DEPLOY_MODE=saas` 环境变量激活
- JWT Token 中携带 `tenantId`

### 4.2 各模块租户隔离状态

#### ✅ 已正确隔离的模块

| 模块 | Controller/Route | 隔离方式 |
|------|-----------------|----------|
| 客户管理 | `customers.ts` | `getTenantRepo(Customer)` + `addTenantFilter(qb)` |
| 部门管理 | `DepartmentController.ts` | `getTenantRepo(Department)` |
| 用户管理 | `UserController.ts` | `getTenantRepo(User)` |
| 订单管理 | `orders.ts` | `getTenantRepo(Order)` |

#### 🔴 未正确隔离的模块

| 模块 | 文件 | 问题 |
|------|------|------|
| **角色管理** | `RoleController.ts` | 原始SQL无 `tenant_id` 过滤 |
| **权限管理** | `PermissionController.ts` | 使用 `AppDataSource.getTreeRepository` 无租户过滤 |
| **角色统计** | `RoleController.getRoleStats` | 全局统计，无租户过滤 |
| **敏感信息权限** | `SensitiveInfoPermission` 实体 | 无 `tenant_id` 字段 |
| **客服权限配置** | `CustomerServicePermission` 实体 | 无 `tenant_id` 字段 |
| **用户个人权限** | `UserPermission` 实体 | 无 `tenant_id` 字段 |

## 五、前端页面级权限控制分析

### 5.1 路由守卫权限检查

| 检查类型 | meta字段 | 判断逻辑 | 问题 |
|----------|---------|----------|------|
| 需要登录 | `requiresAuth` | 检查 `token` 存在 | ✅ 正常 |
| 需要管理员 | `requiresAdmin` | `isAdmin` 计算属性 | ⚠️ 硬编码角色名 |
| 需要经理 | `requiresManager` | `isManager` 计算属性 | ⚠️ 硬编码角色名 |
| 需要超管 | `requiresSuperAdmin` | `isSuperAdmin` 计算属性 | ⚠️ 硬编码角色名 |
| 特殊权限 | `requiresSpecialPermission` | 多条件OR判断 | 🔴 大量硬编码 |
| 权限代码 | `permissions: ['xxx']` | **未在路由守卫中检查！** | 🔴 配置了但未生效 |

> 🔴 **重要发现**：路由 `meta.permissions` 字段（如 `permissions: ['finance:cod']`）在路由守卫中**完全没有被检查**！这意味着这些权限配置形同虚设。

### 5.2 前端菜单渲染与权限

菜单的显示/隐藏应该根据权限控制，但当前路由守卫只检查了角色级别（admin/manager/superAdmin），没有检查细粒度的 `permissions` 数组。

### 5.3 按钮级权限（`v-permission` 指令）

文件 `src/directives/` 目录下可能有权限指令，前端通过 `hasPermission()` 函数检查按钮权限。该函数依赖 `userPermissions` 数组，而这个数组的来源见 3.3 节的优先级链。

## 六、数据范围权限控制分析

### 6.1 后端数据范围控制

**客户列表**（`backend/src/routes/customers.ts` 第 60-138 行）：
```
admin/super_admin → 看全部客户 ✅
department_manager → 看本部门成员创建/分配的客户 ✅
其他角色 → 只看自己创建/分配/分享的客户 ✅
```
> ✅ 客户模块数据范围控制实现较好

**问题**：数据范围判断使用硬编码角色名：
```typescript
if (userRole !== 'admin' && userRole !== 'super_admin') { ... }
const isManager = userRole === 'department_manager' || userRole === 'manager';
```
> 🔴 没有使用 `roles.data_scope` 字段来判断，而是硬编码角色编码。
