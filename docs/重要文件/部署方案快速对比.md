# 部署方案快速对比

## 核心答案

### GitHub仓库分离 ≠ 服务器必须分离

```
GitHub（代码管理）              宝塔服务器（运行环境）
─────────────────              ─────────────────────
4个独立仓库                    可以是1台服务器！
├── crm-system        ────┐
├── crm-website       ────┤
├── crm-admin         ────┼──→  同一台服务器
└── crm-app           ────┘     同一个数据库
                                同一个后端服务
```

## 数据库和后端服务共用规则

### ✅ 可以共用的情况

#### 1. 你自己运营（场景1）
```
同一台服务器上的多个站点：
CRM前端 ──┐
官网前端 ──┼──→ 同一个后端API ──→ 同一个数据库
Admin前端 ─┘

原因：
- 都是你自己的服务
- 数据需要统一管理
- 节省资源
```

#### 2. 多租户SaaS模式
```
多个客户的CRM ──→ 同一个后端API ──→ 同一个数据库
                                    （但有租户隔离）

前提条件：
- 必须实现租户隔离（tenant_id）
- 数据库表都要加租户字段
- 后端API要识别租户
```

### ❌ 不能共用的情况

#### 客户私有部署（场景2）
```
客户A的CRM ──→ 客户A的后端 ──→ 客户A的数据库
客户B的CRM ──→ 客户B的后端 ──→ 客户B的数据库

原因：
- 客户要求数据隔离
- 在不同的服务器上
- 安全性要求
```

## 宝塔面板站点配置

### 你自己运营（1台服务器，4个站点）

```
宝塔面板 → 网站
├── 站点1: api.yourdomain.com (Node项目，端口3000)
├── 站点2: crm.yourdomain.com (静态网站，反向代理到3000)
├── 站点3: www.yourdomain.com (静态网站，反向代理到3000)
└── 站点4: admin.yourdomain.com (静态网站，反向代理到3000)

宝塔面板 → 数据库
└── crm_production (所有站点共用)
```

### 客户私有部署（1台服务器，2个站点）

```
宝塔面板 → 网站
├── 站点1: api.客户域名.com (Node项目，端口3000)
└── 站点2: crm.客户域名.com (静态网站，反向代理到3000)

宝塔面板 → 数据库
└── crm_客户名 (客户独立数据库)
```

## 域名配置示例

### 你自己运营

```

├── crm.yourdomain.com      → CRM系统
```

### 客户私有部署

```
客户域名: 客户的域名.com

子域名配置：
├── crm.客户的域名.com      → CRM系统
└── api.客户的域名.com      → 后端API
```

## 成本对比

| 场景 | 服务器 | 数据库 | 后端服务 | 月成本 | 适用场景 |
|------|--------|--------|----------|--------|----------|
| 场景1 | 1台 | 1个共用 | 1个共用 | ¥200 | 你自己运营 |
| 场景2 | 每客户1台 | 每客户1个 | 每客户1个 | 客户承担 | 客户私有部署 |
| 场景3 | 你1台+客户N台 | 你1个+客户N个 | 你1个+客户N个 | ¥200+客户承担 | 混合模式 |

## 推荐方案

### 如果你要自己运营CRM
👉 **使用场景1**
- 1台服务器部署所有站点
- 共用数据库和后端服务
- 成本最低，管理最简单

### 如果客户要私有部署
👉 **使用场景2**
- 每个客户独立服务器
- 独立数据库和后端服务
- 数据完全隔离，安全性高

### 如果两者都有
👉 **使用场景3**
- 你的官网和Admin在你的服务器
- 客户的CRM在客户的服务器
- 灵活性最高

## 关键要点

### GitHub仓库分离的目的
```
✅ 保护商业机密（官网和Admin代码）
✅ 客户只能获取CRM代码
✅ 代码管理更清晰
❌ 不是为了分离服务器
```

### 服务器部署的灵活性
```
✅ 可以在1台服务器部署多个站点
✅ 可以共用数据库和后端服务
✅ 可以根据需求选择部署方式
❌ 不需要为每个仓库单独部署服务器
```

### 本地开发不受影响
```
✅ 继续在同一个目录开发
✅ 所有项目在一起，方便调试
✅ 只是GitHub上分离了仓库
✅ 部署时可以灵活选择
```

## 总结

**核心答案**：
1. GitHub仓库分离 ≠ 服务器必须分离
2. 你自己运营：1台服务器，共用数据库和后端
3. 客户私有部署：每个客户独立服务器和数据库
4. 本地开发：继续在同一个目录，不受影响

**最佳实践**：
- GitHub：分离4个仓库（保护商业机密）
- 本地：统一目录开发（方便调试）
- 部署：根据场景灵活选择（成本和安全平衡）
