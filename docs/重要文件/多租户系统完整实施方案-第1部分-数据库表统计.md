# 多租户系统完整实施方案 - 第1部分：数据库表统计

## 📊 数据库表完整统计

基于对 `database/schema.sql` 和 `backend/database-schema.sql` 的深入分析，CRM系统共有 **39个数据库表**。

---

## 一、系统表（7个）- 不需要tenant_id

这些表是系统级别的，所有租户共享，不需要添加 `tenant_id` 字段。

### 1. system_configs
- **用途**: 系统配置表
- **说明**: 存储系统级别的配置，如系统名称、版本、时区等
- **为什么不需要tenant_id**: 所有租户共享相同的系统配置

### 2. admin_users
- **用途**: Admin后台管理员账户
- **说明**: 管理Admin后台的管理员账号
- **为什么不需要tenant_id**: Admin后台管理员是平台级别的，不属于任何租户

### 3. tenants
- **用途**: 租户表
- **说明**: 存储所有SaaS租户的基本信息
- **为什么不需要tenant_id**: 这是租户表本身，不需要tenant_id

### 4. tenant_settings
- **用途**: 租户配置表
- **说明**: 存储每个租户的个性化配置
- **为什么不需要tenant_id**: 已经有tenant_id作为外键关联租户表

### 5. tenant_packages
- **用途**: 套餐表
- **说明**: 定义不同的套餐类型（基础版、专业版、企业版）
- **为什么不需要tenant_id**: 套餐是平台级别的，所有租户共享

### 6. versions
- **用途**: 版本发布表
- **说明**: 管理CRM系统的版本发布
- **为什么不需要tenant_id**: 版本是平台级别的，所有租户共享

### 7. licenses
- **用途**: 授权表（私有部署）
- **说明**: 管理私有部署客户的授权码
- **为什么不需要tenant_id**: 私有部署客户使用独立数据库，不需要tenant_id

---

## 二、业务表（32个）- 需要添加tenant_id

这些表是租户级别的业务数据，需要添加 `tenant_id` 字段进行数据隔离。

### 用户权限类（7个表）

#### 1. users - 用户表
- **字段**: id, username, password, name, email, phone, role_id, department_id...
- **需要隔离**: ✅ 是
- **原因**: 每个租户有自己的用户
- **唯一索引修改**: `username` → `(tenant_id, username)`

#### 2. roles - 角色表
- **字段**: id, name, code, description, permissions...
- **需要隔离**: ✅ 是
- **原因**: 每个租户可以自定义角色
- **唯一索引修改**: `name` → `(tenant_id, name)`

#### 3. permissions - 权限表
- **字段**: id, name, display_name, resource, action...
- **需要隔离**: ⚠️ 部分（系统权限不需要，自定义权限需要）
- **原因**: 系统预设权限共享，租户自定义权限隔离
- **建议**: 添加 `is_system` 字段，系统权限 tenant_id=NULL

#### 4. user_roles - 用户角色关联表
- **字段**: id, user_id, role_id...
- **需要隔离**: ✅ 是
- **原因**: 关联表跟随主表

#### 5. role_permissions - 角色权限关联表
- **字段**: id, role_id, permission_id...
- **需要隔离**: ✅ 是
- **原因**: 关联表跟随主表

#### 6. departments - 部门表
- **字段**: id, name, code, parent_id, manager_id...
- **需要隔离**: ✅ 是
- **原因**: 每个租户有自己的部门结构

#### 7. authorized_ips - 授权IP表
- **字段**: id, ip_address, user_id...
- **需要隔离**: ✅ 是
- **原因**: 每个租户有自己的IP白名单

## 四、唯一索引修改清单

以下表的唯一索引需要修改为复合索引（tenant_id + 原字段）：

1. **users**: `username` → `(tenant_id, username)`
2. **users**: `email` → `(tenant_id, email)`
3. **roles**: `name` → `(tenant_id, name)`
4. **roles**: `code` → `(tenant_id, code)`
5. **customers**: `customer_code` → `(tenant_id, customer_code)`
6. **products**: `code` → `(tenant_id, code)`
7. **orders**: `order_number` → `(tenant_id, order_number)`
8. **after_sales_services**: `service_number` → `(tenant_id, service_number)`

**注意**: MySQL允许多个NULL值，所以私有部署模式（tenant_id=NULL）不受影响。
