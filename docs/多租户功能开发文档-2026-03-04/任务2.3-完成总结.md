## 🔍 中间件工作流程

### 请求处理流程

```
客户端请求
    ↓
1. tenantAuth 中间件
   - 验证 JWT Token
   - 提取 tenantId 和 userId
   - SaaS模式：验证租户状态
   - 将租户信息注入 req 对象
    ↓
2. tenantContextMiddleware 中间件
   - 从 req 对象提取租户信息
   - 使用 AsyncLocalStorage 设置上下文
   - 让后续代码可以获取租户信息
    ↓
3. 业务路由处理
   - BaseRepository 自动获取租户ID
   - 自动添加 tenant_id 过滤
   - 数据完全隔离
    ↓
返回响应
```

### 中间件顺序

**关键点**：中间件顺序必须正确
```typescript
app.use(`${API_PREFIX}/customers`, 
  tenantAuth,              // 第1步：认证
  tenantContextMiddleware, // 第2步：设置上下文
  customerRoutes           // 第3步：业务逻辑
);
```

## 🎓 技术要点

### 1. 中间件应用模式

Express中间件按顺序执行：
```typescript
app.use(path, middleware1, middleware2, routeHandler);
```

执行顺序：
1. middleware1 执行
2. middleware2 执行
3. routeHandler 执行

### 2. AsyncLocalStorage 上下文传递

tenantContextMiddleware 使用 AsyncLocalStorage：
```typescript
asyncLocalStorage.run(context, () => {
  next(); // 在上下文中执行后续中间件
});
```

好处：
- 不需要在每个函数中传递 tenantId
- 自动在异步调用链中传递
- BaseRepository 可以直接获取

### 3. 公开路由排除

某些路由不需要租户认证：
- `/auth` - 登录接口（还没有Token）
- `/public` - 公开接口（不需要登录）
- `/calls/webhook` - 第三方回调（外部系统）

### 4. 中间件顺序的重要性

错误的顺序：
```typescript
// ❌ 错误：上下文在认证之前
app.use(path, tenantContextMiddleware, tenantAuth, routes);
```

正确的顺序：
```typescript
// ✅ 正确：先认证，后设置上下文
app.use(path, tenantAuth, tenantContextMiddleware, routes);
```

## ⚠️ 重要注意事项

### 1. 向后兼容

私有部署模式完全兼容：
- tenantAuth 在私有模式下不验证租户
- tenantContextMiddleware 设置 tenantId = undefined
- BaseRepository 自动处理（WHERE tenant_id IS NULL）

### 2. 性能影响

中间件对性能的影响：
- tenantAuth：1次Token验证 + 1次数据库查询（SaaS模式）
- tenantContextMiddleware：仅设置上下文（无数据库操作）
- 总体影响：< 10ms（可接受）

### 3. 错误处理

中间件会自动处理错误：
- Token无效 → 401 Unauthorized
- 租户不存在 → 403 Forbidden
- 租户已禁用 → 403 Forbidden
- 租户已过期 → 403 Forbidden

### 4. 调试建议

如果遇到问题：
1. 检查Token是否包含tenantId（SaaS模式）
2. 检查中间件顺序是否正确
3. 检查路由是否应用了中间件
4. 使用测试脚本验证
