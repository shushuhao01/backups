## 一、项目全景

### 1.1 产品定位

云客CRM是一个 **SaaS + 私有部署双模式** 的完整CRM商业产品体系，包含5个子系统：

| 子系统 | 技术栈 | 代码量 | 文件数 | 端口 | 定位 |
|--------|--------|--------|--------|------|------|
| **Backend** | Node.js + Express + TypeORM + MySQL | 62,647行 | 249文件 | 3000 | 统一后端API服务 |
| **CRM前端** | Vue 3 + TS + Element Plus + Pinia | ~215,000行(含数据文件) | 386文件 | 5173 | 主业务系统（员工使用） |
| **Admin后台** | Vue 3 + TS + Element Plus + Axios | 16,372行 | 39文件 | 5174 | 平台管理后台（运营方使用） |
| **Website官网** | Vue 3 + TS + 纯手写SCSS | 6,838行 | 19文件 | 8080 | 产品展示 + 注册 + 支付 |
| **crmAPP** | uni-app + Vue 3 + TS | 9,001行 | 34文件 | — | 电销外呼手机APP |

**📊 全项目代码总量：约 310,000+ 行（含地址数据文件约55,000行）**

### 1.2 商业闭环

```
潜在客户访问官网 → 选择套餐 → 注册/支付 → 自动开通租户 → 下发登录信息
                                                                     ↓
管理后台运营管理 ← 授权/续费/模块控制 ← 管理员 ← 分配账号 ← 登录CRM系统
                                                              ↓
                                                     CRM日常使用（客户/订单/物流/业绩）
                                                              ↓
                                                     移动端APP → 外呼/录音/跟进
```

### 2.2 部署模式

| 模式 | 环境变量 | 说明 |
|------|---------|------|
| **私有部署** | `DEPLOY_MODE=private` | 单租户，授权码激活，数据tenant_id=NULL |
| **SaaS云端** | `DEPLOY_MODE=saas` | 多租户，共享数据库，tenant_id隔离 |

通过 `backend/src/config/deploy.ts` 统一管理，`BaseRepository` 自动根据模式添加租户过滤。

### 2.3 通信方式

| 连接类型 | 技术 | 用途 |
|---------|------|------|
| HTTP REST | Express + JWT | 所有CRUD操作 |
| WebSocket (Socket.IO) | socket.io | PC端实时消息推送 |
| WebSocket (ws) | ws | 移动端通话指令/状态同步 |
| 定时任务 | node-cron | 授权过期检查、支付提醒、消息清理 |

### 3.5 认证与安全体系

| 层级 | 机制 | 说明 |

### 3.7 后端代码质量评价

#### ✅ 做得好的
1. **错误处理体系完善**：自定义AppError层级 + 全局errorHandler中间件 + 优雅关闭
2. **安全措施全面**：Helmet/CORS/Rate Limiting/登录锁定/IP白名单/CSP策略
3. **日志系统专业**：Winston分级日志、文件轮转（20MB/14天）、Morgan HTTP日志
4. **WebSocket设计优秀**：Socket.IO(PC)+ws(移动)双通道、认证中间件、心跳检测
5. **TypeScript实体规范**：76个实体定义清晰，字段注释完整
6. **租户架构完整**：AsyncLocalStorage + BaseRepository + 中间件，三层联动
7. **定时任务体系**：node-cron调度器 + 5个自动化任务

#### ⚠️ 需要改进的
1. **巨型路由文件**：orders.ts(3400行)、system.ts(2937行)、customers.ts(2016行)
2. **分层不一致**：Admin有Controller→Service模式，但CRM路由直接写业务逻辑
3. **大量代码重复**：时间格式化、权限过滤在每个路由文件重复实现
4. **请求校验不完整**：仅部分路由使用Joi/express-validator
5. **内存缓存风险**：CacheService用Map实现，多实例部署下失效
6. **console.log残留**：大量业务代码使用console而非Winston logger
7. **双密码加密方案**：bcryptjs和pbkdf2+salt并存（历史原因）

### A6.2 客户数据权限过滤逻辑

```typescript
// 后端 customers.ts 权限过滤核心逻辑：
if (角色 === 'admin' || 角色 === 'super_admin') {
    // 管理员：查看全部客户（除非指定onlyMine=true）
} else if (角色 === 'department_manager') {
    // 部门经理：查看本部门 + 分享给自己的客户
    WHERE (sales_person_id IN (部门成员IDs) OR id IN (分享客户IDs))
} else {
    // 普通员工：查看自己创建的 + 分配给自己的 + 分享给自己的
    WHERE (created_by = 当前用户 OR sales_person_id = 当前用户 OR id IN (分享客户IDs))
}
```

### A6.3 代收货款(COD)业务流程

```
创建订单(含COD金额) → 发货 → 快递公司代收 → 已签收
    ↓
两条路径：
  A. 正常流程：财务确认返款 → cod_status='returned' → 返款金额入账
  B. 取消代收：销售员申请 → 管理员审核 → cod_status='cancelled'
```

### A6.4 绩效计算逻辑

```
1. 订单完成时，自动计算绩效：
   绩效金额 = 订单实付金额 × 绩效系数(默认1.0)

2. 佣金计算（支持阶梯提成）：
   佣金 = 绩效金额 × 提成比例
   阶梯示例：0-5万=5%, 5-10万=8%, 10万+=10%

3. 定时任务：每天统计前日绩效数据，生成报表
```
