# 租户数据隔离问题 - 完整修复总结

## 问题回顾

### 用户反馈
用户登录"欢乐颂"租户后，能看到"天河佳信"租户的数据（客户、订单、商品等），说明数据隔离没有生效。

### 初步诊断
运行 `backend/check-tenant-data-isolation.js` 发现：
- 数据库层面：数据隔离正确
  - "欢乐颂"租户：0个客户、0个订单
  - "天河佳信"租户：6个客户、20个订单
- 所有数据都正确标记了 `tenant_id`

### 问题定位
数据库数据正确，但前端能看到其他租户的数据，说明问题在后端API层面。

## 根本原因

### 技术分析

系统使用 AsyncLocalStorage 来传递租户上下文，但在实际执行中出现了"上下文传递断裂"：

1. **tenantContextMiddleware**（在所有路由之前执行）
   - 初始化 AsyncLocalStorage
   - 但此时 `req.tenantId` 还是 `undefined`

2. **tenantAuth**（在各个路由内部执行）
   - 从 JWT token 中提取 `tenantId`
   - 设置 `req.tenantId`
   - ❌ **但没有调用 `TenantContextManager.setContext()` 更新 AsyncLocalStorage**

3. **BaseRepository**（查询数据时）
   - 调用 `TenantContextManager.getTenantId()` 获取租户ID
   - ❌ **获取到的是 `undefined`（因为 AsyncLocalStorage 没有被更新）**
   - 没有添加 `tenant_id` 过滤条件
   - 返回所有租户的数据

### 执行流程图

```
请求
  ↓
tenantContextMiddleware (初始化AsyncLocalStorage，但req.tenantId=undefined)
  ↓
路由
  ↓
tenantAuth (设置req.tenantId，但没有更新AsyncLocalStorage) ❌
  ↓
Controller
  ↓
BaseRepository (从AsyncLocalStorage获取tenantId，获取到undefined) ❌
  ↓
查询数据库（没有tenant_id过滤，返回所有数据） ❌
```

## 修复方案

### 修改文件
`backend/src/middleware/tenantAuth.ts`

### 修改内容

1. **导入 TenantContextManager**
```typescript
import { TenantContextManager } from '../utils/tenantContext'
```

2. **在 SaaS 模式分支中，添加上下文更新**
```typescript
// 更新 TenantContext（关键！让 BaseRepository 能获取到 tenantId）
TenantContextManager.setContext({
  tenantId: req.tenantId,
  userId: req.userId,
  tenantInfo: req.tenantInfo
})
```

3. **在私有部署模式分支中，也添加上下文更新**
```typescript
// 更新 TenantContext
TenantContextManager.setContext({
  tenantId: req.tenantId,
  userId: req.userId
})
```

### 修复后的执行流程

```
请求
  ↓
tenantContextMiddleware (初始化AsyncLocalStorage)
  ↓
路由
  ↓
tenantAuth (设置req.tenantId，并调用TenantContextManager.setContext()更新AsyncLocalStorage) ✅
  ↓
Controller
  ↓
BaseRepository (从AsyncLocalStorage获取tenantId，成功获取到租户ID) ✅
  ↓
查询数据库（添加tenant_id过滤，只返回当前租户的数据） ✅
```

## 技术要点

### AsyncLocalStorage 的使用
AsyncLocalStorage 是 Node.js 提供的异步上下文传递机制，类似于 Java 的 ThreadLocal。它允许在异步调用链中传递上下文数据，而不需要显式地在每个函数参数中传递。

### 关键点
1. **初始化**：`tenantContextMiddleware` 使用 `asyncLocalStorage.run()` 初始化上下文
2. **更新**：`tenantAuth` 使用 `TenantContextManager.setContext()` 更新上下文
3. **读取**：`BaseRepository` 使用 `TenantContextManager.getTenantId()` 读取上下文

### 注意事项
- AsyncLocalStorage 的上下文是在 `asyncLocalStorage.run()` 中创建的
- 在 `run()` 回调函数内部，可以通过 `getStore()` 获取上下文
- 上下文会自动传递到所有异步调用（Promise、async/await、setTimeout等）
- 必须在 `run()` 回调内部更新上下文，否则更新不会生效

## 总结

这是一个典型的"上下文传递断裂"问题。虽然设计上使用了 AsyncLocalStorage 来传递租户上下文，但在实际执行中，`tenantAuth` 中间件没有正确更新 AsyncLocalStorage，导致 `BaseRepository` 无法获取租户ID，从而无法过滤数据。

修复后，租户数据隔离功能将正常工作，每个租户只能看到自己的数据。这是多租户SaaS系统的核心安全功能，确保了数据的隔离性和安全性。
